# Zenn 記事フォーマットスキル

Zennリポジトリ `articles/` 配下のMarkdown記事を、統一された体裁に揃えるスキルです。時期が違う記事に蓄積した表記ゆらぎを、機械的なルールで直します。

## 対象ファイルの決定

ユーザー入力: $ARGUMENTS

- `$ARGUMENTS` がファイルパス（例: `articles/xxxx.md`）なら、そのファイル1つだけを対象にする
- `$ARGUMENTS` が空なら、`Glob` で `articles/*.md` を全件取得し、対象にする
- 対象ファイルが複数件の場合、一気に全部直す前に **対象ファイル数をユーザーに伝え、続行して良いか確認してから** 処理を始めてください。意図しない大量コミットを防ぐためです

## 実行の大まかな流れ

1. 対象ファイル一覧を決める（上記）
2. 各ファイルを `Read` して、以下のルールに沿って必要な修正を `Edit` で適用する
3. 全ファイル処理後、変更があったファイル一覧と、各ファイルで適用したルールを一言で添えて報告する

ファイルは **1ファイルずつ順番に** 処理してください。1ファイル内の複数ルールは、副作用を避けるため「frontmatter → 見出しレベル → コードブロック言語 → 本文スタイル」の順で適用します。

---

## 重要な前提: コードブロックを変換しない

**一番大事なこと**: Zenn記事内のコードブロック（```` ``` ```` で囲まれた範囲）とインラインコード（`` ` `` で囲まれた範囲）は、**どのルールでも絶対に書き換えない** でください。中身の全角文字、句読点、ピリオドなどは原文のまま保存します。

理由: コードの意味が壊れる・ハイライトが崩れる・URLが壊れる、などの副作用が非常に起こりやすく、記事の信頼性を失うからです。

Editするときは置換対象が本文側であることを、周辺コンテキスト（前後の行）で必ず確認してください。1つのEditで広範囲を置換せず、ピンポイントで確実に直してください。

## ルール1: frontmatterの整え

Zenn記事の先頭 `---` で囲まれたYAMLブロックに対して適用します。

### 1-1. キー並び順を固定する

順序: `title → emoji → type → topics → published → published_at → publication_name`

欠けているキーは飛ばす。余分なキー（上記以外）が現れた場合は、`publication_name` の下に元の並び順を保って後置してください。

**変換例:**

```diff
 ---
-published: true
 title: "Goの話"
 emoji: "🐹"
+type: "tech"
 topics: ["go"]
-type: "tech"
+published: true
 ---
```

### 1-2. topics を小文字化する

`topics` 配列の各要素を小文字化する。ただし:
- 日本語（ひらがな・カタカナ・漢字）を含む要素は変更しない（例: `パフォーマンス`）
- 既に小文字のものは変更しない

**変換例:**

```diff
-topics: ["Go", "AWS", "パフォーマンス"]
+topics: ["go", "aws", "パフォーマンス"]
```

### 1-3. 必須キーの欠落を検出（自動補完はしない）

必須キー: `title`, `emoji`, `type`, `topics`, `published`

いずれかが空文字/空配列/未定義なら、最終レポートで「この記事のこのキーが空」と報告してください。勝手に値を埋めないこと。ユーザーが書きかけの状態を尊重します。

### 1-4. 既存のクォートスタイルは触らない

`"..."` / `'...'` / クォートなし、の選択は既存のまま残す。揃えるために置き換えないでください。 `title:` が `"foo"` なら `"foo"`、`'foo'` なら `'foo'` のまま。

---

## ルール2: 見出しレベル（`##` から始める）

Zenn記事はアクセシビリティ上、本文の最上位見出しを `##`（h2）から始める慣習があります。記事タイトルは frontmatter の `title` が担います。

### 2-1. `#` 始まりの記事を検出

コードブロック外で行頭が `# ` (半角スペース1つ + 見出し文) になっている行があるなら、**その記事内の全ての見出しを1段下げる**。

- `#` → `##`
- `##` → `###`
- `###` → `####`
- （以下同様）

コードブロック内（```` ``` ```` の内側）の `#` はコメント行等の可能性が高いので、**絶対に下げない**。

### 2-2. すでに `##` から始まる記事は変更しない

最上位が `##` 以深のみで構成されている記事には何もしない。

**変換例:**

```diff
-# イテレータについて完全理解する
+## イテレータについて完全理解する
 ...
-## 使い方
+### 使い方
```

---

## ルール3: コードブロックの言語指定

言語指定なしの ```` ``` ```` を検出し、中身から言語を推測して付与します。

### 3-1. 検出対象

行頭が厳密に ```` ``` ```` で、その直後に言語識別子も `:ファイル名` もない空のコードフェンス開始行が対象です。既に ```` ```go ```` や ```` ```go:main.go ```` のようになっているものは触らない。

### 3-2. 推測のヒント

中身を見て、以下の手がかりで判定してください。複数ヒットした場合は最も特徴的なものを選択:

| 推測 | ヒント例 |
|---|---|
| `go` | `package `, `func `, `import (`, `:=`, `fmt.` |
| `bash` / `sh` | 先頭が `$ `, `sudo `, `brew `, `npm ` / `npx `, `export `, シェル変数 `$FOO` |
| `json` | 先頭/末尾が `{` または `[`、`"key": ` パターン多発、末尾が `}` `]` |
| `yaml` | 行頭が `- `/`key:` で値、Zenn frontmatterに似る |
| `toml` | `[section]` + `key = value` |
| `python` | `def `, `import `, `print(`, インデントでブロック |
| `javascript` / `typescript` | `const `, `=>`, `import ... from`, `interface `, `type X =` |
| `rust` | `fn `, `let mut`, `::`, `impl ` |
| `diff` | 行頭 `+` `-` `@@` が多い |
| `markdown` / `md` | 見出し `#`、リスト `-`、リンク `[text](url)` が多い |
| `html` | `<tag>` と `</tag>` |
| `sql` | `SELECT `, `FROM `, `WHERE ` (大文字小文字問わず) |

判定できない場合は `text` を付与。ログ（ツール実行結果など）も `text` で良いです。

### 3-3. 変換例

    ```
    package main

    func main() {
        fmt.Println("hi")
    }
    ```

↓

    ```go
    package main

    func main() {
        fmt.Println("hi")
    }
    ```

## ルール4: 本文スタイルの統一

**このルールは本文のみに適用**。以下は対象外:
- コードブロック内（```` ``` ```` の内側）
- インラインコード内（`` `...` `` の内側）
- frontmatter（先頭の `---` で囲まれた範囲）
- URL・リンク記法の URL 側（`[text](URL)` の URL、`![alt](URL)` の URL）
- 引用ブロック（行頭 `>` で始まる行）内

### 4-1. ですます調に統一

文末が「だ」「である」で終わる断定文、および「〜した」「〜する」の地の文を「です」「ます」などに置き換える。

**判断の指針:**
- 体言止め（名詞で終わる短い見出し的な行）は変更しない
- 箇条書きの要素で、文ではなくキーワード/フレーズのものは変更しない
- 引用中の「である調」は、原著者の文体なので変更しない
- 文末に確信がないとき、無理に変換するより元のままにする（過剰変換を避ける）

**変換例:**

```diff
-Goは高速な言語だ。
+Goは高速な言語です。

-このライブラリを使うと便利である。
+このライブラリを使うと便利です。

-並行処理を実装した。
+並行処理を実装しました。
```

体言止めや箇条書きは触らない:

```
## 結論
- Goは速い
- 並行処理が楽
```

### 4-2. 句読点を「、」「。」に統一

日本語文中の句読点を日本語向けに揃える:
- `,` (半角カンマ) → `、`
- `，` (全角カンマ) → `、`
- `.` (半角ピリオドで文末) → `。`
- `．` (全角ピリオド) → `。`

**対象外:**
- コードブロック/インラインコード内
- URL内
- 英語の文章（半角英数のみで構成される1文）の句読点
- 小数点やバージョン番号（`1.2.3`、`0.5`）の `.`
- ファイル拡張子（`.go`、`.md`）の `.`
- ドット記法（`fmt.Println` 等）の `.`
- 略語のピリオド（`e.g.`、`i.e.`）

判断に迷ったら変換しない。過剰変換のほうが害が大きいです。

**変換例:**

```diff
-Goはシンプルだ, と言われています.
+Goはシンプルだ、と言われています。
```

### 4-3. 全角アルファベット・数字を半角に

本文中の全角英数字（`Ａ-Ｚ`, `ａ-ｚ`, `０-９`）を半角に変換。

**対象外:**
- コードブロック/インラインコード内
- URL内
- 絵文字・記号は変換しない（全角スペース `　` はそのまま、句読点は上のルール4-2で扱う）

**変換例:**

```diff
-ＡＷＳ上でＧｏ１．２３を動かす
+AWS上でGo1.23を動かす
```

## 最終レポート

全ファイル処理後、以下の形式で1回だけ報告してください:

```
## フォーマット結果

### 変更したファイル
- articles/xxxx.md
  - ルール2: 見出しレベルを1段下げ（# → ##）
  - ルール3: 3箇所のコードブロックにgo/bash/jsonを付与
- articles/yyyy.md
  - ルール1-2: topics "Go", "AWS" を小文字化
  - ルール4-2: 句読点「,」「.」を「、」「。」に置換（8箇所）

### 変更なし
- articles/zzzz.md (N件 など、件数のみで良い)

### 要確認（自動補完しなかった項目）
- articles/aaaa.md: frontmatter の `emoji` が空欄
```

変更なしのファイルが多い場合は件数サマリで構いません。

## 実装上の注意

- `Edit` の `old_string` が複数箇所にマッチする場合は、`replace_all` を使うか、周辺コンテキストを含めて一意になるようにしてください
- 大きな記事でも、1ファイル = 複数 Edit で分けて修正して構いません。1つのEditで全文置換するのは避ける（差分が追えない、事故る）
- 記事の意味内容には一切手を入れない。見出しの語順入れ替え、段落の整形、誤字修正などはスコープ外です
- 不明な点や、どちらのルールを適用すべきか迷った箇所があれば、修正せず最終レポートで「要確認」として報告してください
