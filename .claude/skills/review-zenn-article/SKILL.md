# Zenn 記事レビュースキル

Zennリポジトリ `articles/` 配下のMarkdown記事に対して、公開前に読者視点で一通り指摘を出すスキルです。体裁の機械的な統一は姉妹スキル `format` の担当なので、このスキルは **人間のレビュアーが読んで気になる観点** に集中します。

## 対象ファイルの決定

ユーザー入力: $ARGUMENTS

優先順位は以下の通りです。

1. `$ARGUMENTS` がファイルパス（例: `articles/xxxx.md`）なら、それをレビュー対象にする
2. `$ARGUMENTS` が空なら、`Glob` で `articles/*.md` を全件取得し、各ファイルの frontmatter の `published:` を見て `published: false` のものだけに絞る
   - 該当0件: 「公開前の下書きがありません。レビューしたいファイルを指定してください」と返して終了
   - 該当1件: それをそのままレビュー対象にする
   - 該当2件以上: 候補一覧（ファイル名 + `title`）を出して、どれをレビューするかユーザーに聞く

意図していない記事を勝手にレビューして長い出力を返すと邪魔になるため、曖昧な場合は必ず確認してください。

## レビューの進め方

1. 対象ファイルを `Read` で全文読む
2. 以下の5観点（「レビュー観点」セクション）を **頭から尻まで** 通して見る。観点ごとに別読みはしない（同じテキストを何度も読むのは非効率で、前後関係で気づく指摘もあるため）
3. 見つけた指摘を重大度で分類し、「出力フォーマット」に従ってチャットに出す

**大事なこと**: 意味内容に介入しない。このスキルは指摘を **出す** だけで、勝手に Edit で直しにいかない。ユーザーが「じゃあ直して」と言ってから初めて修正に入ってください。レビューと修正を同時にやるとユーザーが差分を追いにくくなります。

## レビュー観点

### 観点1: 文章品質

読者がつまずかずに読み通せるか、を見る観点です。

#### 1-1. ですます調の一貫性

記事全体でですます調（「です」「ます」）か、である調（「だ」「である」）のどちらかに揃っているべきです。章をまたいで混ざっている場合は指摘する。

- 1文だけ混ざっているようなケースは見逃さない
- ただし **引用ブロック内** は原著者の文体なので対象外
- 箇条書きの体言止め（「並行処理の基礎」など名詞で終わる項目）は調子の対象外

#### 1-2. 主語と述語の対応

長い文で主語と述語が噛み合っていないと読者は詰まる。典型パターン:

- 主語「〜は」が省略され、誰の話か読者が推測する必要がある
- 主語と述語の間に修飾節が長く挟まり、最後に出てくる述語が主語とかみ合っていない
- 「〜することが〜となる」のような二重名詞化で意味がぼやけている

機械的に検出するのは難しいので、読みながら「あれ、これ誰の話？」と感じた箇所を指摘する。

#### 1-3. 冗長な表現

同じことを複数回言っている、説明が過剰な箇所を指摘する。例:

- 「〜することができます」→ 「〜できます」で十分
- 直前の段落で説明した内容を次の段落でも言い換えて繰り返している
- 「という」「ということ」の多用

ただし **強調のためにあえて繰り返している箇所** は残すべきこともあるため、「この繰り返しは冗長か、意図的な強調か」を文脈で判断する。迷ったら Nit として「気になれば検討」レベルで出す。

#### 1-4. 論理の飛躍

AからBを導く過程で、前提や説明が抜け落ちていないかを見る。典型:

- 「〜なので〜です」と書かれているが、因果関係が読者に伝わらない
- 新しい概念を一言で導入して、その後その概念を既知のものとして扱う
- 結論が書かれているが、その結論を支える根拠が本文中にない

「読者がこの段落で『なんで？』と思うか？」を想像しながら読む。

---

### 観点2: 構成

#### 2-1. 冒頭で「何の記事か」が分かるか

記事の冒頭（最初の2〜3段落、あるいは最初のセクション）で、以下が分かるのが望ましい:

- この記事が何について書かれているか
- 誰向けか（想定読者）
- 読むとどんな疑問が解消されるか、何ができるようになるか

これが欠けていると、読者は「この記事を読むべきか」を判断できず離脱する。冒頭が「きっかけ」の個人的エピソードから始まっていて本題に入るのが遅い場合も指摘対象。

#### 2-2. 見出しの階層

- 最上位見出しは `##` (h2) から始める。`#` で始まる記事は指摘する（記事タイトルは frontmatter の `title` が担うため）
- 見出しレベルが飛んでいないか（`##` の次がいきなり `####` など）
- 見出しの粒度が揃っているか。1章だけ異様に長い、あるいは短すぎる場合は構成の再検討を提案

**注**: 見出しを `#` → `##` に機械的に下げるのは姉妹スキル `format` の仕事。このスキルでは「構成上おかしい」という指摘だけ出し、修正は format に譲る。

#### 2-3. 導入と結論の対応

記事の冒頭で「これについて書きます」と約束したことが、結論で回収されているか。

- 冒頭で提起した疑問に対して、結論が答えていない
- 結論セクションが存在せず、最後の技術セクションで唐突に終わっている
- 結論が冒頭と噛み合わず、論点がすり替わっている

#### 2-4. 読者が迷子にならないか

長い記事で、読者が「今どこを読んでいるか」を見失う構成になっていないかを見る。

- 章と章の接続が説明されておらず、唐突に新しい話題に入る
- 「まずAを説明し、次にBを説明する」という見取り図が冒頭で示されていない（長い記事で特に重要）
- 図や表があると分かりやすくなる箇所を、文章だけで押し通している

---

### 観点3: 技術的正確性

#### 3-1. コード例が動くか

記事に登場するコードスニペットを読んで、以下を確認する:

- 構文エラーで動かない（変数名の typo、閉じ括弧漏れ、セミコロン/カンマ漏れ）
- インポート/import 文が足りない（関数を使っているが from/import が書かれていない）
- API の使い方が古い、誤っている（公式ドキュメントで確認できる範囲で）
- 出力例が記載されているが、そのコードから実際に出る出力と食い違う

自信が持てない場合は「このコードは動作確認しましたか？」と尋ねる形の指摘でもOK。断定しない。

#### 3-2. 用語の誤用

技術用語を本来と違う意味で使っていないか。例:

- 「スレッドセーフ」と「並行安全」が混同されている
- 「同期」と「順次」が入れ替わっている
- 言語固有の概念（Goの goroutine、Rustのライフタイム等）が他言語の類似概念と混同されている

#### 3-3. 事実誤認

検証可能な事実（バージョン番号、リリース日、仕様上の制約、公式ドキュメントに書かれている挙動）が間違っていないか。自信がない場合は「この記述は公式ドキュメントで確認できますか？」と聞く形にする。

---

### 観点4: Zenn 固有のルール

#### 4-1. frontmatter の必須項目

以下が全て揃っていて、空文字/空配列でないこと:

- `title`: 記事タイトル
- `emoji`: 単一絵文字
- `type`: `"tech"` または `"idea"`
- `topics`: タグ配列（空配列 `[]` は不可）
- `published`: boolean

欠けている/空のものがあれば Critical で指摘。

#### 4-2. title と emoji の妥当性

- `title` が空文字、あるいは仮タイトル（「タイトル」「test」など明らかに本番用でないもの）の場合は指摘
- `emoji` は単一絵文字である必要がある。複数絵文字や普通のテキストが入っているなら指摘

#### 4-3. topics の中身

- 記事内容と合致しているか（Goの記事なのに topics に `go` がない、等）
- 大文字小文字の統一は **しない**。ユーザーの表記をそのまま尊重する（例: ユーザーが `Go` と書いていれば `go` に勝手に直す提案はしない。ただし機械的な統一は `format` スキルの担当）

#### 4-4. 見出しレベル

最上位が `#` で始まっている場合は指摘（詳細は 2-2 参照）。

#### 4-5. コードブロックの言語指定

言語指定のない ```` ``` ```` だけのコードフェンスを検出したら指摘。中身から推測できる言語を Nit レベルで提案する（シンタックスハイライトが効かず読みにくくなるため）。

---

### 観点5: 読者視点

#### 5-1. 前提知識の説明

想定読者に対して、前提知識の説明が過不足なく足りているか。

- 想定読者がおそらく知らないであろう用語/概念を、説明なしに使っている → 指摘（一言の脚注や、公式ドキュメントへのリンクがあると親切）
- 逆に、想定読者なら当然知っている内容を長々と説明している → 冗長として指摘（観点1-3と重複することもある）

想定読者は記事の冒頭や `topics` から推測する。「Goの記事で topics に `go` がある」なら、Goの基本文法は説明不要と考える。

#### 5-2. 具体例の有無

抽象的な説明だけで具体例が出てこない箇所を指摘する。「例えば...」「たとえば...」があると読者の理解が一段深くなる。

#### 5-3. 「なぜこれを学ぶか」の動機づけ

技術記事では、導入部分で「なぜこれを知ると嬉しいのか」が読者に伝わっていると読み進めてもらえる。知識の単なる列挙になっていないかを見る。

## 重大度の分類

指摘を以下の3段階に分類してください。

### Critical

- **公開したら記事の信頼性が損なわれる**レベル
- 例: 事実誤認、動かないコード、frontmatter の必須項目欠落、用語の明確な誤用、結論が冒頭の主張と矛盾している

### Important

- **読者体験を明確に損なう**が、致命的ではないレベル
- 例: 章をまたいだですます調混在、冒頭で記事の目的が分からない、論理の飛躍で読者が詰まる箇所、見出しレベルの破綻、想定読者が絶対知らない用語の未説明

### Nit

- **より良くするための提案**。無視しても記事は成立する
- 例: 「〜することができます」の冗長さ、より簡潔な言い回しの提案、コードブロックの言語指定漏れ、具体例を足すと良い箇所

指摘が1つもない観点があっても構いません。無理に水増ししないでください。逆に、同じ箇所に対する指摘は観点が複数にまたがっていても1つにまとめて構いません（読者が読みやすい）。

## 出力フォーマット

レビュー完了後、以下の形式で **1回だけ** チャットに出力してください。**指摘は必ず1つの表にまとめる**（重大度ごとに見出しを作って箇条書きで並べない）。ダラダラと縦に伸びると、どの観点の指摘がどれだけ重いのかを読者が一瞥で把握できず、レビュー結果として使いにくくなるためです。

```
## レビュー結果: articles/xxxxxx.md

> タイトル: "記事のタイトル"
> 想定読者: topicsなどから推測して1行で

### 指摘一覧 (Critical: N件 / Important: N件 / Nit: N件)

| #   | 優先度     | 観点           | 該当箇所                   | 指摘                                       | 修正案                                         |
| --- | ---------- | -------------- | -------------------------- | ------------------------------------------ | ---------------------------------------------- |
| 1   | Critical   | 3-1 コード例   | L120 の `main` 関数        | `fmt` パッケージが import されていない     | 冒頭に `import "fmt"` を追加                   |
| 2   | Important  | 2-1 冒頭       | 「## この記事で知れること」| 1行のみで情報量が薄い                      | 本文で扱う項目に合わせて4-5行に拡充 (※1)       |
| 3   | Nit        | 1-3 冗長       | L14                        | 「解説したものになります」が回りくどい     | 「解説します」に短縮                            |

※1 具体的には以下のような項目を列挙する:
- Go 1.24 以前からある「容量定数 make」の条件
- Go 1.25 の 32 バイトバッファ方式
- Go 1.26 の `runtime.move2heap()` の仕組み

### 総評
- **強み**: この記事の良いところを1-2点
- **方向性**: 全体として改善するとより良くなるポイントを1-2点
```

### 各カラムの書き方

- **#**: 通し番号。**優先度順** に振る（Critical → Important → Nit）。優先度が変わるタイミングで見出しを挟まず、1つの表に全件入れる
- **優先度**: `Critical` / `Important` / `Nit` の3段階（絵文字や記号は付けない）
- **観点**: `3-1 コード例` / `2-1 冒頭` のように、**観点番号 + 2〜6文字の短いラベル** を入れる。番号は「レビュー観点」セクションのものと一致させる
- **該当箇所**: `L42` / `L120-L140` / `「## 見出し名」セクション` など、読者がエディタで一発で辿れる形。「全体的に」だけで済ませない
- **指摘**: 何が問題かを **1文で** 書く。背景説明や前置きは書かない
- **修正案**: どう直すかを **1文で**、可能なら before/after の形で書く。セルに収まらないほど長くなる場合は `(※1)` のように脚注参照にし、表の直後に「※1」ブロックを置く

### 件数サマリ

表の見出しに `(Critical: N件 / Important: N件 / Nit: N件)` を付ける。0件のものも `Critical: 0件` と **素直に明記する**（省略しない）。「Critical が 0 件であること」自体が読者にとって重要な情報だからです。

### 指摘がゼロのとき

表ごと省略し、「指摘なし。公開準備できています」と1行で返して、総評だけ添える。空の表は出さない。

### 総評

- 2〜3行で収める。長い総評は総評ではなく「表に入れ損ねた指摘」になりがち
- 「強み」と「方向性」の2軸に絞る。それ以上の項目を増やさない

### 表にしない例外

指摘が **1件のみ** で、かつ修正案が表のセルに到底収まらない場合（例: 記事の構成そのものの再編提案）は、表ではなく見出し+段落で書いて構わない。ただし迷ったら表にする。表にして困ることはほとんどない

## やらないこと

- Edit で勝手に修正しない（ユーザーが修正を依頼してから初めて着手）
- 体裁の機械的統一（句読点、全角半角、見出しレベルの下げ等）→ `format` スキルの担当なので、「format スキルで直せる範囲です」と一言添えて指摘に留める
- 記事の主張自体を否定する批評（「この意見は間違っている」等）。意見記事の価値観には介入しない。あくまで「読者に伝わるか」「技術的に正しいか」の観点に絞る
- 記事が長大で全部を細かく見るとキリがない場合でも、少なくとも各観点1回は通して目を通す。読み飛ばさない
