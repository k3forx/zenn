# Zenn テックブログ記事生成スキル

このスキルの肝は「**実行時にお気に入り上位5記事を読んで真似る**」ことです。
固定のスタイルルールをこのファイルに書き込むと、ユーザーの書き方が進化したときに古くなります。だから毎回、実際の記事をサンプルとして取り直します。

## 入力

ユーザー入力: $ARGUMENTS

- 引数あり: トピック・技術名として使用
- 引数なし: `AskUserQuestion` でトピックを入力

## 実行フロー

### Phase 1: トピックを決める

1. `$ARGUMENTS` が空なら AskUserQuestion で聞く
2. 空文字/100文字超は警告

### Phase 2: お気に入り上位5記事を取得して全文読む

**これを飛ばすと、このスキルの意味がなくなります。** 必ず実行してください。

#### 2-1. Zenn API で上位5件を取得

```
WebFetch:
  url: https://zenn.dev/api/articles?username=kkkxxx&order=liked_count
  prompt: "Return top 5 articles as JSON with title, slug, liked_count sorted by liked_count desc."
```

失敗したら、以下の既知の上位5件をフォールバックとして使う（2026-04-19時点）:

| 順位 | slug | タイトル |
|----|------|---------|
| 1 | `2c11335be49f84` | go test の33個のフラグを全て解説してみた！！！ |
| 2 | `a95f084bcd9f88` | テックブログの書き方、をブログにしてみた |
| 3 | `0b57bba3a3665f` | Go1.22で登場するかも！？開発中の新機能イテレータを使ってみる！ |
| 4 | `d9505540581b5d` | Go1.23のイテレータを完全理解する✌️ |
| 5 | `d1a14c19359eec` | Go1.22からfor loopの挙動が変わるかも！？ |

#### 2-2. 5記事をローカルから全文読む

```
各 slug について:
  Read: ~/repos/zenn/articles/{slug}.md
```

大きい記事もあるので、`Read` で必要なら複数回に分けて**全文**読む。Front Matterだけ読んで済ませない。

`Read` の結果に `PARTIAL view — showing lines 1-N of M total` のような打ち切り警告（続きを `offset` 付きで読むよう促す system-reminder）が付いていたら、**次の記事に進む前に**その記事を `offset` 指定で最後まで読み切る。最上位の記事ほど後半に本題（検証・結果・まとめ）が来ることが多く、後半を落とすと文体・論理展開を模倣するというこのスキルの前提が崩れるため。

#### 2-3. 5記事から以下を把握する（自分の頭の中で整理）

- **Front Matter の傾向**: emoji の選び方（定型より気分優先）、topics の粒度、`published_at` や `publication_name` の有無
- **導入の入り方**: 公式ブログ引用→意訳パターン / 問いかけパターン / 結論先出しパターンなど
- **章構成**: 頻出する章タイトル（「この記事で知れること」「結論」「事前準備」「まとめ」など）とその並び順。見出しの記号自体は Phase 3 で `##` 始まりに統一するので、ここでは「どんな章が、どの順で並ぶか」を見る
- **文体**: ですます調の密度、「！」の頻度、絵文字の配置、読者への語りかけ
- **論理展開**: 事実 → 説明 → 検証 → 結果の流れが実際どう現れているか
- **コードの書き方**: 実コードの量、bash実行結果の貼り方、`:::message` や `:::details` の使い方
- **引用翻訳の形式**: `> 英文` のあとの「(意訳)」の書き方

### Phase 3: 記事を書く

Phase 2 で読んだ5記事を **few-shot example** として、新規記事を書きます。以下を守ってください。

#### 文体・構成について

- **固定テンプレに従わない**。Phase 2 で読んだ記事が第一の参考資料。このSKILL.mdに書かれていない癖があっても、記事から読み取ったパターンを優先する
- **プレースホルダーだらけの骨組みで終わらせない**。実コード・実ターミナル出力・実データをできる限り書き込む。トピックに関する公式ドキュメントのURLが分かっているなら、引用文（英語なら意訳付き）を実際に書く
- **見出しは `##` から始める**。最上位を h2（`##`）、その下を `###`。サンプル記事が `#` 始まりでも真似ず `##` 始まりに統一する（アクセシビリティ方針。既存記事の `#` は format スキルが1段下げる対象）。階層の深さの感覚だけサンプルに合わせる
- **「！」「絵文字」は模倣しつつ乱用しない**。ユーザーは適度に使う
- **箇条書きはAI生成感を出さない**。1つのリストは3項目前後に絞り、括弧書きの多用・疑問文風の項目・不自然に形式を揃えた並びを避ける

#### Front Matter のルール

```yaml
---
title: "<記事タイトル>"
emoji: "<1文字絵文字>"
type: "tech"   # ほぼ常にtech。思考・エッセイ系なら idea
topics:
  - "Go"       # 2-4個。ユーザー入力の表記を尊重し、Go→go のような小文字化はしない
  - "..."
published: false
---
```

- **`publication_name` は書かない**。ユーザーは最近これを外す方針（個人記事として公開）。古い記事には残っているが、新規では不要
- **`published_at` も基本的には書かない**（公開時に決める）。下書き段階では `published: false` のみ
- **emoji は内容に合う1文字**。既存記事は 🚀 ⚡ ✏️ 😎 🥷 🤌 🤩 🤖 🔧 🪑 など多様。定型に縛られない

#### 導入の3パターン（観察された型）

どれかを採用する。トピックに合わせて選ぶ:

**A: 公式ブログ/ドキュメント引用型**（#3, #4, #5で使用）

```markdown
## 導入

{技術/バージョン}の[{公式ブログタイトル}]({URL})を読んでいたら、以下の記述が目に留まりました。

> {英文引用}

(意訳)
{日本語訳}

{この記事を書こうと思った動機・疑問}
```

**B: 問いかけ型**（#1で使用）

```markdown
## 導入

みなさん、{読者が遭遇しそうな場面}はありますか？
{自然な導線で本題へ}
```

**C: 結論先出し型**（#5で使用）

```markdown
## 導入
{簡潔な状況説明}

## この記事で知れること
- ...

## 結論
- ...
```

#### よく使われる章タイトル

状況に応じて：「導入」「この記事で知れること」「結論」「事前準備」「仕様を見てみる」「さっそくコードを書いてみる」「{概念}について」「まとめ」

### Phase 4: zenn-cli で新規記事を作成

**ファイル作成は `zenn-cli` スキルを使う**（`Skill` ツールで `zenn-cli` を起動、または同スキルの手順に従う）。自前で `npx zenn` を叩かず、バージョンチェック → `new:article` の流れをそのスキルに任せることで二重メンテを避ける。

そのうえで、このスキルからは次を守る:

```bash
cd ~/repos/zenn
# --machine-readable で、生成された articles/<slug>.md のパスだけを取得する
CREATED=$(npx zenn new:article --title "<タイトル>" --type tech --emoji <絵文字> --machine-readable)
```

- `--slug` は付けない（自動生成に任せる。手書き・改名は禁止）
- `$CREATED`（＝ `articles/<slug>.md`）を `Write` ツールで、生成した Front Matter + 本文で上書きする
- `ls -t articles/*.md | head -1` で最新ファイルを推測する方法は取り違えの恐れがあるので使わない

### Phase 5: 報告

以下を明示:
- 生成ファイルのパス
- タイトル、emoji、topics
- 参考にした上位5記事のタイトル（どの記事から何を取り入れたか1行でよい）
- 次にやると良いこと: `npx zenn preview` / エディタで開く / コード例を実装

## エッジケース

| ケース | 対応 |
|--------|------|
| $ARGUMENTS が空 | AskUserQuestion |
| 100文字超 | 短縮を提案 |
| Zenn API失敗 | フォールバック5件を使う |
| フォールバックのローカルファイルもない | `Glob: ~/repos/zenn/articles/*.md` して最大5件を読む |
| `~/repos/zenn/articles/` がない | エラー表示、パス確認を促す |
| zenn-cli失敗 | まず `zenn-cli` スキルのバージョンチェック/更新を試す。それでも失敗するなら手書き slug は避け、原因（パス・権限）をユーザーに報告して指示を仰ぐ |

## アンチパターン（やってはいけないこと）

- **5記事を読まずに書く**: このSKILL.md内の記述だけで生成すると文体が古くなる。Phase 2 は必須
- **プレースホルダー（{TODO}、`// TODO`）を大量に残す**: ユーザーが埋める前提でも、書ける範囲は書き込む
- **画一的な構成を適用**: 「導入 → 〜とは → 実装 → ベンチ → まとめ」を全記事に押し付けない。トピックによって最適な構成は違う
- **`publication_name` を書く**: 現方針では不要
- **絵文字を連発**: 各段落に絵文字、は過剰
