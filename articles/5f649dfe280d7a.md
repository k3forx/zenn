## 導入

ここ最近、**ハーネスエンジニアリング (Harness Engineering)** という言葉をよく見かけるようになりました。

AIコーディングエージェントをどう「うまく働かせるか」という文脈で、各社・各論者がこぞって語っているテーマです。一方で、誰が言い始めたのかも曖昧で、定義も主張も結構バラバラ、というのが正直な現状だったりします（このあたりは後で軽く触れます）。

そんな中で、Martin Fowler の [Harness engineering for coding agent users](https://martinfowler.com/articles/harness-engineering.html) を読んでいたら、以下の記述が目に留まりました。

> Not every codebase is equally amenable to harnessing.

(意訳)
すべてのコードベースが、等しくハーネスしやすいわけではない。

そして Fowler は、その「ハーネスしやすさ」を決める因子の1つとして **「強い静的型を持つこと」** を挙げています。

これを読んだとき、Goエンジニアである自分は素朴にこう思いました。

**「Goという言語の気質は、ハーネスエンジニアリングという営みとそもそも相性が良いのでは？ でも、ダメな部分もあるのでは？」**

この記事では、具体的な実装やコードには踏み込まず、**ハーネスを「4つの象限」に分解したうえで、各象限にGoの言語的・仕様的な性質を当てはめ、相性を考察** してみたいと思います。良い相性もあれば、Goでもうまくハマらない部分もある、という両面から見ていきます。

:::message
ハーネスエンジニアリングは発展途上の概念で、論者によって定義が異なります。この記事はあくまで2026年中頃時点の議論を踏まえた、筆者なりの整理と考察です。事実と個人的な見解はなるべく分けて書くようにしています。
:::

## この記事で知れること

- ハーネスを「役割 × 性質」の4象限に分解した整理
- 各象限に、Goのどの言語的・仕様的な性質が効くのか／効かないのか
- 結果として **Goは「型・仕様で表現できる範囲」と相性が良く、できない範囲では負荷が別象限へ逆流する** という考察

逆に、**具体的なコード例や実装** は書きません。あくまで言語の性質レベルの話に絞ります。各社のハーネスエンジニアリングの定義の細かい比較も、良い資料が多いので記事末尾の参考リンクに譲ります。

## ハーネスを4象限に分解する

考察の土台を用意します。前提は、Addy Osmani がブログで紹介している Viv Trivedy の以下のシンプルな式です。

> Agent = Model + Harness. If you're not the model, you're the harness.

つまり、**モデル (LLM) 以外のすべてが harness** です。そのうえで、Fowler はハーネスを **調速機 (cybernetic governor)** にたとえ、2つの要素の組み合わせとして捉えています。

> The agent harness acts like a cybernetic governor, combining feed-forward and feedback to regulate the codebase towards its desired state.

(意訳)
エージェントハーネスは調速機のように振る舞い、フィードフォワードとフィードバックを組み合わせて、コードベースを望ましい状態へと制御する。

この **役割 (Guides / Sensors)** と **性質 (Computational / Inferential)** を掛け合わせると、ハーネスは次の4象限に分解できます。本記事は、この4象限を章立ての背骨にします。

| | **Computational**（決定論的・速い・確実） | **Inferential**（確率的・遅い・高コスト） |
| ---- | ---- | ---- |
| **Guides**（動く前に方向づける） | ① 初期化スクリプト、codemod、`go generate` | ② AGENTS.md / CLAUDE.md、skills、規約 |
| **Sensors**（動いた後に観測・修正させる） | ③ 型チェック、gofmt、vet、test、`-race` | ④ AIコードレビュー、意味的チェック、評価器 |

そして、このモデルでいちばん大事な原則がこれです。

> the earlier you find issues, the cheaper they are to fix

(意訳)
問題を早く見つけるほど、修正コストは安くなる。

**安くて速くて確実な計算的要素（左側）をできるだけ手前に寄せ、推論的要素（右側）は本当に必要なところだけに使う** のが良いハーネス、という話です。

各象限を採点するときのモノサシも、先に4つ宣言しておきます。

- **left-shift**：安い計算的要素をどれだけ手前に置けるか
- **agent-legibility**：センサーの出力をエージェントが読めるか
- **型での不変条件表現**：正しさをどこまでコンパイラ側に押し込めるか
- **前提の安定性**：仕様や知識が古びないか

では、象限ごとにGoを当てていきます。

## 象限① 計算的ガイド × Go

まずは「動く前に、決定論的に方向づける」象限です。初期化スクリプト、codemod、コード生成などが入ります。

正直に言うと、**ここはGoの言語仕様が直接効く部分が薄い** 象限です。bootstrap スクリプトやセットアップは、言語というよりプロジェクト個別の作り込みだからです。

そのうえで、Goが標準で差し込めるものとしては **`go generate`** があります。コード生成という計算的ガイドが言語ツールに組み込まれているのは小さな利点ですし、`gofmt -w` のように「書いたそばから決定論的に正準形へ寄せる」仕組みも、広い意味での計算的ガイドと言えます（gofmt 自体は本質的には後述する象限③の計算的センサーですが、`-w` で保存時に自動整形する運用ではガイド的にも働きます）。

- **相性：○（ただし言語の貢献は限定的）**
- ここは「Goだから強い」というより「どの言語でも設計者が作り込む」領域。深追いはしません。

## 象限② 推論的ガイド × Go

次は「動く前に、確率的に方向づける」象限です。AGENTS.md / CLAUDE.md、skills、コーディング規約ドキュメントなどが入ります。エージェントが読んで従う（とは限らない）類のガイドです。

ここでGoの「気質」が **間接的に、しかし大きく効きます**。

### 追い風：仕様が小さく、やり方が1つだから、書くべきガイドが減る

Goは言語仕様が小さく直交的で、「やり方は基本1つ」を good とする文化を持ちます。gofmt が整形を一意に決め、明示性を重んじる設計のため、**「インデントは」「import 順は」「この書き方とあの書き方どっち」といった規約を、推論的ガイドに延々と書く必要がありません**。

推論的ガイドは、書くほど長くなり、長いほどエージェントが守りきれなくなる、という弱点を抱えています。Goは **そもそもガイドに書くべき項目を減らしてくれる** ので、この象限を軽く保てます。これは地味ですが大きな相性の良さです。

### 逆風：型で書けない不変条件が、この象限に「逆流」してくる

一方で、後述する象限③（計算的センサー）でGoの型が表現できない不変条件——「この値は3択のいずれか」「switch は全ケースを処理せよ」「JSONタグはこのフィールドに対応」——は、**行き場を失って、この推論的ガイドに逆流してきます**。

「列挙は必ず全部 handle してね」と AGENTS.md に書く、という対応は、まさに **本来コンパイラに任せたい不変条件を、いちばん壊れやすい推論的ガイドに押し付けている** 状態です。Goの型の表現力の天井が、そのままこの象限の負担増として現れる、というのは見落としやすい逆風だと思います。

- **相性：○（追い風）と △（逆風）が同居**
- Goは「スタイル系のガイド」を不要にする一方、「型で書けない不変条件のガイド」をここに溜めてしまいます。

## 象限③ 計算的センサー × Go（本丸）

「動いた後に、決定論的に観測して直させる」象限です。型チェック、gofmt、go vet、golangci-lint、go test、`-race` などが入ります。**Goがいちばん輝く本丸であり、同時に最も鋭い弱点も抱える** 象限です。

### 追い風1：強い静的型 — 言語に内蔵された決定論的センサー

最も基本的な計算的センサーが「型チェック」で、Goはこれを言語そのものに内蔵しています。型エラーは **決定論的な「事実」** であり、同じコードなら必ず同じ結果を返し、モデルが賢くなっても見逃しません。left-shift の理想形であり、Fowler の因子その1をそのまま満たします。

### 追い風2：gofmt — スタイルの非決定性を仕様で消す

スタイルは本来「正解が複数ある」非決定論的な論点ですが、Goは単一の正準フォーマットに固定することで、**この非決定性を仕様レベルで丸ごと消して** います。象限②のガイドを軽くする効果も、元をたどればここです。

### 追い風3：ツールチェーンの一体性と速さ

go vet・go test が標準同梱で、コンパイルが速い。**センサーが最初から手元にあり、回す待ち時間も短い** ので、エージェントのループの内側に計算的センサーを組み込みやすい。「早期検出は安い」を構造的に実践できます。`-race` という、型では捕まえられないデータ競合のための専用センサーまで標準で用意されています。

ここまでがGoのほぼフルマークのA面です。**ここからがB面、つまり「強い静的型なのに守れない範囲」** です。

### 逆風1：sum型・網羅性チェックの不在 — 型の天井

Goには代数的データ型 (sum型) も exhaustiveness チェックもなく、`enum` は `iota` ＋ただの `int` です。`switch` に新しいケースを足し忘れてもコンパイルは通り、3択であるべき値に任意の int を渡せてしまいます。

つまり **「全ケースを処理せよ」「値はこのN択」という不変条件を、型で表現できない**。これは他の強い型の言語（Rust、TypeScript の判別共用体など）なら計算的センサーで弾ける類のバグが、Goではすり抜けることを意味します。**「強い静的型◎」の最大の例外** であり、`exhaustive` のような linter を別途足して埋める必要があります。

### 逆風2：zero値 / nil の罠 — 「型は通るのに壊れている」

Goは未初期化フィールドを黙って zero 値にします。nil map への書き込みは実行時 panic、typed-nil を入れた interface は `!= nil` と判定され、nil ポインタ参照も実行時 panic——**どれもコンパイルエラーになりません**。

zero 値設計は「未初期化エラーが存在しない」代わりに「黙って誤ったデフォルトで動く」状態を許します。**コンパイルが通る・でも意味的に壊れたコード** をエージェントが量産しうる、という点で、計算的センサーの守備範囲の外に大きな穴が空いています。

### 逆風3：struct tag + reflection の標準慣習

イディオムな JSON / シリアライズは、`json:"name"` のような **文字列タグ＋実行時 reflection** で実現されます。タグのタイポやマッピングの型不一致はコンパイル時に捕まらず、実行時に黙ってミスマップ、あるいは失敗します（`go vet` はタグの構文は検査しますが、フィールドとの意味的対応までは見ません）。

標準的な書き方そのものが、**正しさを型システムの外（実行時の、文字列頼みの "stringly-typed" な世界）へ追い出している** わけです。

- **相性：◎（型・gofmt・ツール・速さ）と ×△（sum型・zero値・struct tag）が同居**
- Goの計算的センサーは極めて強力ですが、**型で表現できる範囲には明確な天井がある**。天井を越える不変条件は、linter追加で埋めるか、象限②④へ逃がすことになります。

## 象限④ 推論的センサー × Go

最後は「動いた後に、確率的に観測して直させる」象限です。AIコードレビュー、意味的チェック、評価器 (AI as Judge) などが入ります。

### 追い風：可読性が、レビューする側のエージェントを助ける

OpenAI はハーネスにおいて、コードを **agent-legible（エージェントが読める形）** にすることを重視しています。

> Humans steer. Agents execute.

(意訳)
人間は操縦し、エージェントが実行する。

Goの「明示性」「読みやすさ最優先」「やり方は1つ」という思想は、**レビューする側のエージェントにとっての読みやすさ** に直結します。隠れた制御フローや暗黙の変換が少ないぶん、推論的センサーが対象コードを誤読しにくい。これは入力側の追い風です。

### 逆風1：意味的な正しさは、この象限に頼るしかない

象限③で見たとおり、型が保証するのは「型」までです。仕様上「合計」を返すべき関数が「積」を返していても型は通ります。**意味的な正しさは計算的センサーの守備範囲外** なので、テスト設計やAIレビューといった、この象限の推論的センサーに頼らざるを得ません。Goの型が強いからといって、この負荷が減るわけではないのです。

### 逆風2：標準エラーにスタックトレースがない — legibility の低下

Goの標準の `error` は **スタックトレースを持ちません**（`panic` やテスト失敗・レース検出は発生箇所を出しますが、通常のエラー値は「どこで起きたか」を標準では運びません）。

これは推論的センサーの **agent-legibility** に効きます。センサーやログが `connection refused` とだけ返して呼び出し経路が分からないと、エージェントは原因箇所を特定しづらい。**フィードバック信号の質が、標準のままだと低い** わけです。`fmt.Errorf("...: %w", err)` でのラップ規約や、ログにスタックトレースを含める計装で補う必要があり、これはちょうど[オブザーバビリティ向上の取り組み](https://zenn.dev/canary_techblog/articles/dc2ad12cf2d78e)で `error.stack` をログに足したような、設計者の仕事として残ります。

- **相性：○（可読性）と ×△（意味的正しさ・legibility）が同居**
- 象限③で型に書けなかったものは、結局この象限へ流れ込みます。**Goは「計算的に弾けない正しさ」の負荷を、推論的センサーに集約させる傾向があります**。

## 4象限まとめ

ここまでの考察を、象限ごとに表にまとめます。

| 象限 | 効くGoの性質（相性◎○） | 効かない／逆風（△×） |
| ---- | ---- | ---- |
| ① 計算的ガイド | `go generate`、gofmt -w（○・限定的） | 言語仕様の貢献は薄い |
| ② 推論的ガイド | 小さい仕様・gofmt がガイドを軽くする（○） | 型で書けない不変条件が逆流（△） |
| ③ 計算的センサー | 強い静的型・gofmt・vet/test・速さ・-race（◎） | sum型/網羅性なし(×)、zero値/nil(△×)、struct tag(△) |
| ④ 推論的センサー | 明示性・可読性が agent-legible（○） | 意味的正しさの集約(×)、stacktrace無しでlegibility低下(△×) |

象限③に◎が集中し、その天井を越えた不変条件が ②と④ に△×として逆流している、という構図が見えます。

## 横断的な追い風：仕様の安定性

4象限のどこにも属さないけれど、全体に効く性質として **Go 1 後方互換性の約束** を挙げておきます。

ハーネスもモデルも、ある時点の言語仕様・API を前提に動いています。仕様が頻繁に壊れると、AIの知識も組んだハーネスもすぐ古びてしまいます。Goは仕様の安定に強くコミットしているので、**4象限すべての前提が相対的に陳腐化しにくい**。これは Anthropic の次の見解とも響き合います。

> the space of interesting harness combinations doesn't shrink as models improve. Instead, it moves

(意訳)
面白いハーネスの組み合わせの余地は、モデルが賢くなっても縮まない。むしろ移動するだけだ。

ここから先は筆者の解釈ですが、ハーネスが「移動」していくとしても、**土台の言語仕様が安定していれば、その移動のコストは小さく済む** はずです。地味ですが、長い目で見ると効いてくる相性の良さだと思います。

## 考察の結論

4象限に当てて見えてきたのは、はっきりした非対称性です。

**Goは「型・仕様で表現できる範囲」のハーネス要素、とりわけ象限③（計算的センサー）と圧倒的に相性が良い** です。強い静的型・gofmt・一体化したツールチェーン・安定した仕様は、どれも「安くて速くて確実な計算的要素」そのもの。さらに **「明示性」「可読性」「やり方は1つ」というGoの設計思想自体が、象限②を軽くし、象限④を読みやすくする** 方向に働きます。ハーネスという言葉が生まれる前から、Goは Mitchell Hashimoto の言う「同じ間違いを二度させない仕組み」を、gofmt や vet という形で実践してきた言語だとも言えます。

一方で、**「型・仕様で表現できない範囲」——sum型で表すべき不変条件、zero値が覆い隠す誤状態、並行性、意味的な正しさ、エラーの可観測性——は、Goでも計算的に弾けません**。そしてそれらは消えるのではなく、**象限③から②・④へと「逆流」し、推論的ガイドや推論的センサーの負荷として残ります**。

つまり、**Goは「計算的に守れる領域を広く取れる言語」であって、「ハーネスを不要にしてくれる言語」ではない**、というのが今回の考察の結論です。Goを選ぶことはハーネスの土台（特に左下の象限）を強くしますが、型に書けないものをどう右側の象限で受け止めるかは、やはり自分たちで設計する必要がある、ということですね。

## まとめ

- ハーネスは **役割(Guides/Sensors) × 性質(Computational/Inferential)** の4象限に分解でき、原則は **安い計算的要素を手前に寄せること**
- Goは **象限③（計算的センサー）** と圧倒的に相性が良いです。強い静的型・gofmt・一体化ツール・安定仕様が効きます
- **象限②（推論的ガイド）** はGoの小さい仕様によって軽くなる一方、型で書けない不変条件の逃げ場にもなります
- **型・仕様で表現できない範囲**（sum型/網羅性、zero値、並行性、意味的正しさ、stacktrace無し）は計算的に弾けず、**②・④へ逆流** します
- Goは"ハーネスしやすい"が"ハーネス不要"ではありません。Goの気質を活かして左下を固め、右側で受け止める設計を考えるのが良さそうです

Goを書いている人にとって、ハーネスエンジニアリングは決して遠い概念ではありません。**普段当たり前に使っている型チェックや gofmt、go vet を「象限③の計算的センサー」として捉え直す** だけでも、見え方が変わってくるのではないでしょうか。
