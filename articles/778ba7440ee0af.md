## 導入

前回、Claude Code のスキル（`SKILL.md`）が使われるたびに自分で改善案を出してくる、という自己改善ループを作った話を書きました。

https://zenn.dev/kkkxxx/articles/cd55956dffa373

このとき、記事の中で **eval（スキルの評価）は一旦スコープ外** とはっきり書いていました。仕組みは作ったものの、「このループは本当に効いているのか？」を一度も測っていなかったのです。

チームへの展開を進める前に、まずは個人ループが本当に機能しているのかを確かめたくなりました。そこで、退避されたログを実際に集計して「直っているのか」をデータで見てみたところ、**「修正を当てているのに、エラー率がまったく下がらない」という一見おかしな結果** に行き当たりました。

この記事は、その正体を追いかけた記録です。結論を先に言うと、**skill-fix は壊れていませんでした。ただ、見ていた指標がまずかっただけ** でした。

:::message
この記事の数値はすべて 2026-07-08 時点の `~/.claude/skill-logs/` を集計したものです。後述しますが、スキルの実行回数の分母が構造的に欠損するため、発生率はすべて概算です。
:::

## この記事で知れること

- 自己改善ループが「実際に何件直したのか」の測り方
- 「エラー」と「曖昧な指示」で、修正の効き方が真逆になること
- 直す機構を賢くするより先に、効果測定の土台を作るべきだという教訓

## skill-fix とはどんなスキルか

この記事の主役は、前回作った `skill-fix` というスキルです。ひとことで言うと **「他のスキルを直すためのスキル（メタスキル）」** で、`/skill-fix` で起動します。

`skill-fix` は、スキルが使われるたびに溜まっていく「フリクションログ」を、実際の `SKILL.md` の修正に変換します。全体は「収集（全自動）」と「消化（人間の承認つき）」の2段階ループになっています。

1. **収集**: スキルを使ったセッションの終わりに `SessionEnd` フックがログをキューに積み、次のセッション開始時に headless の Claude がそれを「蒸留」して、各スキルの `## エラー箇所` と `## 曖昧だった指示`（＋改善案）を `~/.claude/skill-logs/` に書き出します
2. **消化**: `/skill-fix` を実行すると、溜まったログをスキル単位に束ね、各改善案を **現行の `SKILL.md` を正として** サブエージェント（verifier）が検証し、`apply` / `already-fixed` / `needs-redesign` / `reject` に分類します
3. **適用**: `apply` になった修正だけを diff として提示し、人間が承認したら適用します。処理済みログは `processed/` に退避し、記録を `applied.jsonl` に残します

肝は **「収集は全自動・適用は人間の承認つき」** という非対称です。ログを集めるのはコストがかからないほど良く、スキルの書き換えは間違うと毎回の実行に影響するので、そこだけ人間の承認を挟んでいます。

前回はここまでを「作った」話でした。今回は、この `skill-fix` が本当に効いているのかを、退避された処理済みログから測ってみます。

## まず退避ログを数えてみる

まずは `~/.claude/skill-logs/processed/` に溜まった「処理済み」の山を数えるところから始めます。

集計すると、退避ログは **92 件**、そこから抽出された finding は **計 139 件** でした。内訳は `## エラー箇所` が 65 件、`## 曖昧だった指示` が 74 件と、ほぼ拮抗しています。対象スキルは 18 個、期間は 6/26〜7/6 です。

スキル別に finding を積み上げると、こんな分布になっていました。

![スキル別 finding 内訳](/images/778ba7440ee0af/skill-finding-breakdown.png)
_退避ログから抽出した finding を、スキル別に「エラー箇所 / 曖昧だった指示」で積み上げたもの_

件数（ログの数）で並べると、上位は PR・CI まわりの支援スキルでした。

| 順位 | スキル                    | ログ件数 |
| ---- | ------------------------- | -------- |
| 1    | `draft-pr`                | 11       |
| 2    | `watch-ci`                | 11       |
| 3    | `address-review-comments` | 9        |

よく使うスキルほどログが多い、というごく自然な結果に見えます。ただ、この「絶対数」の順位は、あとで正規化するとひっくり返ります。

## 「エラー箇所」と「曖昧だった指示」とは何か

finding は大きく2種類に分かれます。この2つの違いが、この記事の結論に直結するので先に定義しておきます。前回の蒸留プロンプトが実際に使っている見出しが、そのまま2つのカテゴリになっています。

- **エラー箇所**: そのセッションで現実に起きた実行時の失敗。発生源の多くは外部環境・権限・CLI 仕様です。
  - 例: HTTP 403、権限拒否、存在しない `--host` オプション、`jq` が空を返す、asdf の `No version` エラー
- **曖昧だった指示**: 失敗ではなく、スキルの記述が足りずモデルが即興で対応した箇所。本文の内在的な欠陥です。
  - 例: 実行予算が未定義、手順が未記載、キー名が固定的すぎる、省略可否が不明確

ざっくり言うと、**エラーは「外で起きた事故」、曖昧は「スキル本文の書き足りなさ」** です 🧐 この区別が、あとで効いてきます。

## 件数と発生率で、順位が逆転する

先ほどの「件数ランキング」には落とし穴があります。`draft-pr` は 200 回近く使っていて 11 件、`agent-device` は十数回しか使っていないのに同じくらいの件数、という具合に、**分母（使用回数）がバラバラ** だからです。

そこで、推定実行回数で正規化して「1 実行あたりのフリクション発生率」を出してみます。

| スキル           | 推定実行 | 発生率 |
| ---------------- | -------- | ------ |
| `agent-device`   | 約12回   | 83%    |
| `dispatch-goals` | 約7回    | 71%    |
| `watch-ci`       | 約133回  | 12%    |
| `draft-pr`       | 約200回  | 9.5%   |

件数トップだった `draft-pr`・`watch-ci` が、発生率で見ると最も低い側に来ました。つまり **「ログが多い＝壊れている」ではない** ということです。件数は使用頻度に引っ張られるだけで、健全さの指標にはなりません。

:::message
前述のとおり、実行回数の分母は正確には取れない（cron やヘッドレス実行はトランスクリプトが残らない）ため、ここの発生率も概算として見てください。
:::

## 本題：直しているのに、なぜエラーは下がらないのか

ここからが本題です。

まず確認したのは「そもそも skill-fix はサボっていないか？」でした。適用記録の `applied.jsonl` を見ると、**78 レコード中 71 件が applied**。ちゃんと修正は当たっています。

修正が効いているなら、発生率は下がっているはずです。そこで対話系スキル7件を集約し、各スキルの初回修正日で「修正前」「修正後」に分けて発生率を比べてみました。

| 種別         | 修正前 | 修正後 |
| ------------ | ------ | ------ |
| エラー発生率 | 5%     | 13%    |
| 曖昧発生率   | 51%    | 20%    |

![修正前後の発生率](/images/778ba7440ee0af/before-after.png)
_エラーは下がらず、曖昧だけが半分以下になる（対話系スキル7件の集約・概算）_

はっきりと非対称でした。**曖昧な指示は半分以下に減った一方、エラーは下がらない**（むしろ見かけ上は上がっている）のです。

:::message
修正前はサンプルが数日ぶんしかなく、エラーの「上昇」は前の値がたまたま低かった影響を含みます。要点は「上がった」ではなく「下がらない」の方です。
:::

## なぜエラーだけ下がらないのか

理由は、さきほどの定義に立ち返ると腑に落ちます。

- **発生源が本文の外にある**。403・権限拒否・存在しない CLI オプションは、`SKILL.md` の文言を直しても発生自体は止まりません。修正は「起きた後の回避策」の追記にとどまります。
- **母集団が入れ替わる**。同じ `agent-device` でも、非 ASCII 失敗 → lease 競合 → スクショ取得エラー と、時期ごとに別のエラーが出ます。1つ潰すと次が出る moving target です。
- **権限系は管轄外**。権限フックの追記は skill-fix の verifier が reject します（`update-config` 経由でユーザーが決める領域だからです）。

逆に、曖昧な指示は本文の内在的な欠陥なので、1〜2 行の明確化で実際に減ります。前回の記事で「`review-reminder` の出力のぶれが直った」という例を出しましたが、あれはまさに「曖昧だった指示」を潰したケースでした。今回のデータは、それが `review-reminder` 単体の偶然ではなく、**曖昧カテゴリ全体で起きている** ことを裏づけています。

だから **「曖昧は下がり、エラーは残る」** という非対称が生まれます。言い換えると、**skill-fix は壊れていません。ただ、いま見ている指標（エラー発生数）では効果が見えないだけ** でした。

## 改善案を出したら、レビューに刺された

ここまでで「じゃあ指標を付け替えよう」という改善案が7項目ほど浮かびました。が、勢いで実装する前に、skill-fix の実装を精読した独立レビュアー5人＋統合役に評価してもらったところ、けっこう刺されました。

- 原案7項目のうち、そのまま通ったものは少数。多くは「絞れ」「report-only にしろ」の判定
- **事実誤認が2つ** 見つかりました。たとえば「編集すべきファイル」を取り違えていて、正しくは別のスクリプトの該当箇所でした
- 全体の結論は **「推測ベースの分岐を足す前に、効いているかを測る土台を作れ」**

最も反対が集まったのは「エラーに対して eval を既定で on にする」案でした。狙うエラーの大半は環境依存なので、eval しても信号はゼロでコストだけ増える、という指摘です。

逆に全員一致で「最高レバレッジ」とされたのは、**指標を「発生数」から「未リカバリ率」に付け替える** 案でした。未リカバリ率とは、ざっくり言えば「そのエラーが起きても、最終的にタスクを完了できたか（復帰できたか）」で測る指標です。

自分の提案が半分近く落ちるのは正直こたえましたが、「skill-fix は壊れているのか、それとも指標のアーティファクトなのか」という一番の対立点も、指標を付け替えて初めてデータで決着できる、という整理には納得しました。

## 評価の本を読み直して、verifier 自体も直した

指標の付け替えに進もうとしていたところで、Chip Huyen『AI Engineering』（O'Reilly）の評価に関する章（3〜4章）を読み直しました。すると、**skill-fix 自体が「AI-as-Judge を持つ AI システム」だった** ことに気づきます。修正案を検証する verifier サブエージェントは、まさに本でいう「LLM を評価者に使う（AI-as-Judge）」構成そのものです。

本の評価チェックリストを skill-fix に当てると、5観点レビューにも自分の原案にも無かった層 —— **評価者（verifier）そのものの品質** —— に穴が見つかりました。なかでも効いたのが **冗長性バイアス（verbosity bias）** です。AI-as-Judge は長い応答を高く評価しがちで、さらに自分と同じモデルの出力を優遇しやすい、という性質です（本でも、モデルが自分の出力を高めに評価してしまう例が挙げられています）。

これは、この記事の前半で引っかかっていた謎の答えでもありました。エラーに対する修正は「回避策をドキュメントに **追記** する」＝ **文章が長くなる** 方向のものが多く、しかも verifier は提案側と同じモデル族です。つまり、**効かない追記ほど verifier に通りやすい構造** になっていました。「なぜ発生を止めない追記が apply され続けたのか」の根本原因が、ここにありました 🧐

そこで、本で挙げられている評価の観点と、それを skill-fix にどう活かしたかを表にまとめます。根本原因に近く修正も軽いもの（バイアス対策・temperature 0・判定者バージョンの記録）は先に反映し、残りは中期の宿題としています。

| 本で言われていること                                                                           | skill-fix にあった穴                                                                          | どう活かしたか                                                                                                                             |
| ---------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| **冗長性・自己バイアス**（AI-as-Judge は長い応答や、自分と同じモデルの出力を高く評価しがち）   | verifier が提案側と同じモデル族で、エラー修正は「追記＝長くなる」。効かない追記ほど通っていた | **（反映済み）**`verifier.md` に「happy path を変えない追記は加点しない」「`new_string` が `old_string` を包含するだけの提案を疑う」を明記 |
| **判定者ドリフト**（評価者の LLM 自身が時間で変わり、改善か判定者の変化か区別できない）        | verifier のモデルやプロンプト更新の記録がなく、前後比較の物差しが不定                         | **（反映済み）**適用ログに verifier のモデルと `verifier.md` のハッシュを記録し、比較は同じ判定者設定に固定                                |
| **メタ評価**（評価パイプライン自体の再現性や、判定の一致率を測れ）                             | verifier の再現性も一致率も測っておらず、temperature 指定も無い                               | **（反映済み）**verifier を temperature 0 に固定（同じ finding を2エージェントで検証して一致率を測るのは中期）                             |
| **評価の標準化**（基準は進化してよいが、評価自体はバージョン管理して経時比較できるように）     | 毎回その場のログで場当たり評価しており、固定ベースラインが無い                                | **（中期）**recovered / unrecovered 履歴からスキル別「回帰フリクションセット」を貯め、改変のたびに同じ基準で再実行                         |
| **絶対しきい値**（「A が B に勝つ」は“用途に十分か”までは答えない）                            | 「前より良い」しか見ておらず、到達すべき目標が無い                                            | **（中期）**スキル別に目標未リカバリ率を定め、合否を相対でなく絶対しきい値で判断                                                           |
| **スライス評価**（集約だけ見ず分割し、シンプソンのパラドックスを避けよ）                       | 当初の調査がまさに嵌まり、集約エラー率が per-skill の姿を隠していた                           | **（中期）**保留した発生源分類を「スライス次元」のタグにし、レポートを skill × 種別 × recovered で分割集計                                 |
| **サンプルサイズ**（小さな改善の検出には多数サンプルが要る／bootstrap で暴れるなら信用しない） | `run_harness` の「1 finding × 3 回」は統計的に弱い                                            | **（中期）**小効果のエラー修正は eval に頼らず、実運用サンプルを時間で蓄積して測る                                                         |

面白いのは、本が **「まず測る」という方向は正しい** と裏づけつつ、その一方で「**評価者自体を疑え**」という、レビューにも無かった観点を足してくれたことです。verifier まわりの穴は根本原因に近く、修正も数行と軽かったので、後述の指標の付け替えより先に、ここだけ手を入れました。

## 結論：効果測定は、直す機構より先に

ここまでを整理すると、skill-fix への打ち手は「評価者（verifier）自身の質」と「測る指標」の2本柱でした。前者（冗長性バイアスなど）は原因がはっきりしていたので先に直し、後者＝指標の付け替えがこれからの本丸です。

今回いちばんの学びはこれです。

**効果を測る土台を作る前に、直す機構のほうを賢くしようとしていました。**

skill-fix は 71 件の修正を当てていて、「曖昧な指示」は目に見えて減っていました。にもかかわらず、素朴に「エラー発生数」を見ていたせいで「効いていない」ように見えてしまっていました。指標が現実に追いついていなかったわけです。（verifier のバイアスのように原因がはっきりしていて修正も軽いものは別ですが、多くの改善案はそうではありませんでした。）

なので次の一手は、機構の追加ではなく **指標の付け替え** から始めます。承認負荷はゼロで、セクション構造も変えず、「エラー修正が効いているか」を初めて可視化する最小の一歩です。エラーログの分類と扱いを before / after で並べると、狙いがはっきりします。

| 観点             | Before（発生数ベース）                      | After（未リカバリベース）                                                    |
| ---------------- | ------------------------------------------- | ---------------------------------------------------------------------------- |
| エラーごとの記録 | 失敗した事実だけ                            | `[recovered]` / `[unrecovered]` を付け、復帰できたかで区別する               |
| 発生源での分類   | 分けていない                                | 環境／権限／**遵守失敗**（指示に従えなかった失敗）などのタグで分ける（中期） |
| 処理の優先度     | エラーの**発生数**（`error_count`）が多い順 | **未リカバリのエラー**（`unrecovered_count`）を最優先                        |
| 「効いた」の判定 | 発生率が下がったか                          | 未リカバリ率が下がったか（復帰できるようになったか）                         |

いちばん効くのは1行目の `[recovered]` / `[unrecovered]` マーカーです。これが付いて初めて、「発生は止められなくても、ちゃんと復帰できるようになったか」という、いまの修正が実際に効いている軸で測れるようになります。

## まとめ

- 自己改善ループを「作る」段階から「測る」段階に進めてみました
- エラーと曖昧な指示では修正の効き方が真逆で、素朴な発生数では効果が見えないことがわかりました
- verifier 自体も AI なので、冗長性バイアスのような「評価者の癖」を疑う必要がありました
- 教訓は「効果測定は、直す機構より先に作れ」。測れないものは、良くなったかどうかも言えません

前回、この仕組みで目指したいのは **使われるほど自分で良くなっていくドキュメント** だと書きました。ただ「良くなっている」と言い切るには、まず良くなったことを測れないといけない、というのが今回の反省です。

次は未リカバリ率を実装して、また続報を書ければと思います。ここまで読んでくださり、ありがとうございました！
