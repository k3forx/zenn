## 導入

前回、Claude Code のスキル（`SKILL.md`）が使われるたびに自分で改善案を出してくる、という自己改善ループを作った話を書きました。

https://zenn.dev/kkkxxx/articles/cd55956dffa373

このとき、記事の中で **eval（スキルの評価）は一旦スコープ外** とはっきり書いていました。仕組みは作ったものの、「このループは本当に効いているのか？」を一度も測っていなかったのです。

チームへの展開を進める前に、まずは個人ループが本当に機能しているのかを確かめたくなりました。そこで、退避されたログを実際に集計して「直っているのか」をデータで見てみたところ、**「修正を当てているのに、エラー率がまったく下がらない」という一見おかしな結果** に行き当たりました。

この記事は、その正体を追いかけた記録です。結論を先に言うと、**skill-fix は壊れていませんでした。ただ、見ていた指標がまずかっただけ** でした。

:::message
この記事の数値は原則 2026-07-08 に `~/.claude/skill-logs/` を集計したものです（後半の「通読編」以降の棚卸しデータのみ 2026-07-15 時点）。finding の内訳は当時退避済みだった処理済みログ 92 件（6/26〜7/6 のセッション分）を、発生率は未処理分も含む全フリクションログ 128 件を対象にしています。後述しますが、スキルの実行回数の分母が構造的に欠損するため、発生率はすべて概算です。
:::

## この記事で知れること

- 自己改善ループが「実際に何件直したのか」の測り方
- 「エラー」と「曖昧な指示」で、修正の効き方が真逆になること
- 直す機構を賢くするより先に、効果測定の土台を作るべきだという教訓
- 『AI Engineering』の評価・データセットの章を、動いている自作パイプラインに突き合わせて棚卸しする手順
- 評価の軸を「北極星・ガードレール・診断」の3層に分ける整理

## skill-fix とはどんなスキルか

この記事の主役は、前回作った `skill-fix` というスキルです。ひとことで言うと **「他のスキルを直すためのスキル（メタスキル）」** で、`/skill-fix` で起動します。

`skill-fix` は、スキルが使われるたびに溜まっていく「フリクションログ」を、実際の `SKILL.md` の修正に変換します。全体は「収集（全自動）」と「消化（人間の承認つき）」の2段階ループになっています。

1. **収集**: スキルを使ったセッションの終わりに `SessionEnd` フックがログをキューに積み、次のセッション開始時に headless の Claude がそれを「蒸留」して、各スキルの `## エラー箇所` と `## 曖昧だった指示`（＋改善案）を `~/.claude/skill-logs/` に書き出します
2. **消化**: `/skill-fix` を実行すると、溜まったログをスキル単位に束ね、各改善案を **現行の `SKILL.md` を正として** サブエージェント（verifier）が検証し、`apply` / `already-fixed` / `needs-redesign` / `reject` に分類します
3. **適用**: `apply` になった修正だけを diff として提示し、人間が承認したら適用します。処理済みログは `processed/` に退避し、記録を `applied.jsonl` に残します

肝は **「収集は全自動・適用は人間の承認つき」** という非対称です。ログを集めるのはコストがかからないほど良く、スキルの書き換えは間違うと毎回の実行に影響するので、そこだけ人間の承認を挟んでいます。

前回はここまでを「作った」話でした。今回は、この `skill-fix` が本当に効いているのかを、退避された処理済みログから測ってみます。

## まず退避ログを数えてみる

まずは `~/.claude/skill-logs/processed/` に溜まった「処理済み」の山を数えるところから始めます。

集計すると、退避ログは **92 件**、そこから抽出された finding は **計 139 件** でした。内訳は `## エラー箇所` が 65 件、`## 曖昧だった指示` が 74 件と、ほぼ拮抗しています。対象スキルは 18 個、期間は 6/26〜7/6 です。

スキル別に finding を積み上げると、こんな分布になっていました。

![スキル別 finding 内訳](/images/778ba7440ee0af/skill-finding-breakdown.png)
_退避ログから抽出した finding を、スキル別に「エラー箇所 / 曖昧だった指示」で積み上げたもの_

件数（ログの数）で並べると、上位は PR・CI まわりの支援スキルでした。

| 順位 | スキル                    | ログ件数 |
| ---- | ------------------------- | -------- |
| 1    | `draft-pr`                | 11       |
| 2    | `watch-ci`                | 11       |

よく使うスキルほどログが多い、というごく自然な結果に見えます。ただ、この「絶対数」の順位は、あとで正規化するとひっくり返ります。

## 「エラー箇所」と「曖昧だった指示」とは何か

finding は大きく2種類に分かれます。この2つの違いが、この記事の結論に直結するので先に定義しておきます。前回の蒸留プロンプトが実際に使っている見出しが、そのまま2つのカテゴリになっています。

- **エラー箇所**: そのセッションで現実に起きた実行時の失敗。発生源の多くは外部環境・権限・CLI 仕様です。
  - 例: HTTP 403、権限拒否、存在しない `--host` オプション、`jq` が空を返す、asdf の `No version` エラー
- **曖昧だった指示**: 失敗ではなく、スキルの記述が足りずモデルが即興で対応した箇所。本文の内在的な欠陥です。
  - 例: 実行予算が未定義、手順が未記載、キー名が固定的すぎる、省略可否が不明確

ざっくり言うと、**エラーは「外で起きた事故」、曖昧は「スキル本文の書き足りなさ」** です 🧐 この区別が、あとで効いてきます。

## 件数と発生率で、順位が逆転する

先ほどの「件数ランキング」には落とし穴があります。`draft-pr` は 200 回近く使っていてフリクションが出たセッションは 19、`agent-device` は十数回しか使っていないのに 10、という具合に、**分母（使用回数）がバラバラ** だからです。

そこで、推定実行回数で正規化して「1 実行あたりのフリクション発生率」を出してみます。分子は「そのスキルでフリクションが記録されたセッション数」で、冒頭に書いたとおり、ここだけ未処理分も含む全フリクションログ 128 件がベースです（先ほどのログ件数表より数字が大きいのはそのためです）。

| スキル           | フリクションセッション | 推定実行 | 発生率 |
| ---------------- | ---------------------- | -------- | ------ |
| `agent-device`   | 10                     | 約12回   | 83%    |
| `dispatch-goals` | 5                      | 約7回    | 71%    |
| `watch-ci`       | 16                     | 約133回  | 12%    |
| `draft-pr`       | 19                     | 約200回  | 9.5%   |

件数トップだった `draft-pr`・`watch-ci` が、発生率で見ると最も低い側に来ました。つまり **「ログが多い＝壊れている」ではない** ということです。件数は使用頻度に引っ張られるだけで、健全さの指標にはなりません。

:::message
分母の「推定実行」は、手元に残っている対話セッションのトランスクリプト約 4,000 件から、スキルの起動シグナル（Skill ツールの呼び出しやスラッシュコマンドなど）を数えた概算です。cron やヘッドレス実行はトランスクリプトが残らず分母から欠けるため、ここの発生率も概算として見てください。
:::

## 本題：直しているのに、なぜエラーは下がらないのか

ここからが本題です。

まず確認したのは「そもそも skill-fix はサボっていないか？」でした。適用記録の `applied.jsonl` を見ると、**78 レコード中 71 件が applied**。ちゃんと修正は当たっています。

修正が効いているなら、発生率は下がっているはずです。そこで、分母を推定できる対話系スキル（cron やヘッドレス専用でないスキル）のうち、修正前後の比較に足るログがある7件を集約し、各スキルの初回修正日で「修正前」「修正後」に分けて発生率を比べてみました。

| 種別         | 修正前 | 修正後 |
| ------------ | ------ | ------ |
| エラー発生率 | 5%     | 13%    |
| 曖昧発生率   | 51%    | 20%    |

![修正前後の発生率](/images/778ba7440ee0af/before-after.png)
_エラーは下がらず、曖昧だけが半分以下になる（対話系スキル7件の集約・概算）_

はっきりと非対称でした。**曖昧な指示は半分以下に減った一方、エラーは下がらない**（むしろ見かけ上は上がっている）のです。

:::message
修正前はサンプルが数日ぶんしかなく、エラーの「上昇」は前の値がたまたま低かった影響を含みます。要点は「上がった」ではなく「下がらない」の方です。
:::

## なぜエラーだけ下がらないのか

理由は、さきほどの定義に立ち返ると腑に落ちます。

- **発生源が本文の外にある**。403・権限拒否・存在しない CLI オプションは、`SKILL.md` の文言を直しても発生自体は止まりません。修正は「起きた後の回避策」の追記にとどまります。
- **母集団が入れ替わる**。同じ `agent-device` でも、非 ASCII 失敗 → lease 競合 → スクショ取得エラー と、時期ごとに別のエラーが出ます。1つ潰すと次が出る moving target です。
- **権限系は管轄外**。権限フックの追記は skill-fix の verifier が reject します（`update-config` 経由でユーザーが決める領域だからです）。

逆に、曖昧な指示は本文の内在的な欠陥なので、1〜2 行の明確化で実際に減ります。前回の記事で「`review-reminder` の出力のぶれが直った」という例を出しましたが、あれはまさに「曖昧だった指示」を潰したケースでした。今回のデータは、それが `review-reminder` 単体の偶然ではなく、**曖昧カテゴリ全体で起きている** ことを裏づけています。

だから **「曖昧は下がり、エラーは残る」** という非対称が生まれます。言い換えると、**skill-fix は壊れていません。ただ、いま見ている指標（エラー発生率）では効果が見えないだけ** でした。

## 改善案を出したら、レビューに刺された

ここまでで「じゃあ指標を付け替えよう」という改善案が7項目ほど浮かびました。が、勢いで実装する前に、skill-fix の実装を精読した独立の5つのレビューエージェント（ワークフロー設計・verifier ロジック・測定・保守性・懐疑派の5レンズ）＋統合役に評価してもらったところ、けっこう刺されました。

- 原案7項目のうち、そのまま通ったものは少数。多くは「絞れ」「report-only にしろ」の判定
- **事実誤認が2つ** 見つかりました。たとえば「編集すべきファイル」を取り違えていて、正しくは別のスクリプトの該当箇所でした
- 全体の結論は **「推測ベースの分岐を足す前に、効いているかを測る土台を作れ」**

skill-fix には、修正案が効くか（finding が再現しなくなるか）をテスト実行で確かめる簡易 eval ハーネス（`run_harness`）があります。最も反対が集まったのは、これを「エラー由来の修正に既定で on にする」案でした。狙うエラーの大半は環境依存なので、eval しても信号はゼロでコストだけ増える、という指摘です。

逆に全員一致で「最高レバレッジ」とされたのは、**指標を「発生数」から「未リカバリ率」に付け替える** 案でした。未リカバリ率とは、ざっくり言えば「そのエラーが起きても、最終的にタスクを完了できたか（復帰できたか）」で測る指標です。

自分の提案の大半が原形をとどめなかったのは正直こたえましたが、「skill-fix は壊れているのか、それとも指標のアーティファクトなのか」という一番の対立点も、指標を付け替えて初めてデータで決着できる、という整理には納得しました。

## 評価の本を読み直して、verifier 自体も直した

指標の付け替えに進もうとしていたところで、Chip Huyen『AI Engineering』（O'Reilly）の評価に関する章（3〜4章）を読み直しました。すると、**skill-fix 自体が「AI-as-Judge を持つ AI システム」だった** ことに気づきます。修正案を検証する verifier サブエージェントは、まさに本でいう「LLM を評価者に使う（AI-as-Judge）」構成そのものです。

本の評価チェックリストを skill-fix に当てると、5観点レビューにも自分の原案にも無かった層 —— **評価者（verifier）そのものの品質** —— に穴が見つかりました。なかでも効いたのが **冗長性バイアス（verbosity bias）** です。AI-as-Judge は長い応答を高く評価しがちで、さらに自分と同じモデルの出力を優遇しやすい、という性質です（本でも、モデルが自分の出力を高めに評価してしまう例が挙げられています）。

これは、この記事の前半で引っかかっていた謎の答えでもありました。エラーに対する修正は「回避策をドキュメントに **追記** する」＝ **文章が長くなる** 方向のものが多く、しかも verifier は提案側と同じモデル族です。つまり、**効かない追記ほど verifier に通りやすい構造** になっていました。「なぜ発生を止めない追記が apply され続けたのか」の根本原因が、ここにありました 🧐

そこで、本で挙げられている評価の観点と、それを skill-fix にどう活かしたかを表にまとめます。根本原因に近く修正も軽いもの（バイアス対策・決定性の明記・判定者バージョンの記録）は先に反映し、残りは中期の宿題としています。

| 本で言われていること                                                                           | skill-fix にあった穴                                                                                                     | どう活かしたか                                                                                                                                                             |
| ---------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **冗長性・自己バイアス**（AI-as-Judge は長い応答や、自分と同じモデルの出力を高く評価しがち）   | verifier が提案側と同じモデル族で、エラー修正は「追記＝長くなる」。効かない追記ほど通っていた                            | **（反映済み）**`verifier.md` に「happy path を変えない追記は加点しない」「`new_string` が `old_string` を包含するだけの提案を疑う」を明記                                 |
| **判定者ドリフト**（評価者の LLM 自身が時間で変わり、改善か判定者の変化か区別できない）        | verifier のモデルやプロンプト更新の記録がなく、前後比較の物差しが不定                                                    | **（反映済み）**適用ログに `verifier.md` の版数ハッシュを自動記録し、verifier のモデルも併記できるようにして、比較は同じ判定者設定に固定                                   |
| **メタ評価**（評価パイプライン自体の再現性や、判定の一致率を測れ）                             | verifier の再現性も一致率も測っておらず、決定性の担保も無い                                                              | **（反映済み）**verifier のプロンプトに「同じ finding には同じ判定を返す」決定性（temperature 0 相当）を明記（同じ finding を2エージェントで検証して一致率を測るのは中期） |
| **評価の標準化**（基準は進化してよいが、評価自体はバージョン管理して経時比較できるように）     | 毎回その場のログで場当たり評価しており、固定ベースラインが無い                                                           | **（中期）**recovered / unrecovered 履歴からスキル別「回帰フリクションセット」を貯め、改変のたびに同じ基準で再実行                                                         |
| **絶対しきい値**（「A が B に勝つ」は“用途に十分か”までは答えない）                            | 「前より良い」しか見ておらず、到達すべき目標が無い                                                                       | **（中期）**スキル別に目標未リカバリ率を定め、合否を相対でなく絶対しきい値で判断                                                                                           |
| **スライス評価**（集約だけ見ず分割し、シンプソンのパラドックスを避けよ）                       | 当初の調査がまさに嵌まり、集約では 5%→13% に見えたエラー率の変化も、スキル別に分割すると分母が極小（1〜2件の変動）だった | **（中期）**保留した発生源分類を「スライス次元」のタグにし、レポートを skill × 種別 × recovered で分割集計                                                                 |
| **サンプルサイズ**（小さな改善の検出には多数サンプルが要る／bootstrap で暴れるなら信用しない） | `run_harness` の「1 finding × 3 回」は統計的に弱い                                                                       | **（中期）**小効果のエラー修正は eval に頼らず、実運用サンプルを時間で蓄積して測る                                                                                         |

面白いのは、本が **「まず測る」という方向は正しい** と裏づけつつ、その一方で「**評価者自体を疑え**」という、レビューにも無かった観点を足してくれたことです。verifier まわりの穴は根本原因に近く、修正も数行と軽かったので、後述の指標の付け替えより先に、ここだけ手を入れました。

## 通読編：3・4・8章と、パイプライン全体を突き合わせる

前節までの verifier の修正は、いわば「本の刺さった箇所をつまみ読みして直した」段階でした。その後、『AI Engineering』の **3章（評価方法論）・4章（AI システムの評価）・8章（データセットエンジニアリング）** を通読して、収集フックから skill-fix までのパイプライン全体を棚卸しし直しました。

観点は章に合わせて3つに分けました。

- **評価方法論（3章）**: フリクションを抽出する分析器（headless Claude）と verifier は、どちらも AI-as-Judge。判定者としての品質は担保できているか
- **評価の適用（4章）**: 評価パイプラインとしての設計。コンポーネント別の評価、指標とゴールの接続、経時比較の物差し
- **訓練データ作成（8章）**: フリクションログは「スキルを直すためのデータセット」とみなせる。品質・カバレッジ・量の3基準を満たしているか

全部で20個ほど穴が見つかりました。代表的なものだけ表にします。

| 観点       | 見つかった穴                                                                                                                     | 本の該当箇所                              |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------- |
| 評価方法論 | 判定者バージョンの記録が verifier 側だけで、**上流の分析器（プロンプト＋モデル）が未追跡**。ここが変わると経時比較の物差しごと変わる | 3.4.3.2（基準の曖昧さ）                   |
| 評価方法論 | 分析器のモデル指定がエイリアスのままで、**モデル世代が上がると判定者が無告知に入れ替わる**                                          | 4.2.3（不透明なモデル更新）               |
| 評価の適用 | **使用回数の分母が構造的に残らない**。この記事の発生率が全部「概算」なのは、これが根本原因                                           | 4.3.3.2（サンプルサイズ・スライス評価）   |
| 評価の適用 | verifier の判定を人間がどれだけ覆したかを記録しておらず、**判定者と人間の一致率が測れない**                                         | 3.4.1（人間との一致率が Judge の信頼根拠） |
| 評価の適用 | eval シナリオの assertion に**修正文の逐語句を書けてしまう**。「挙動の改善」でなく「追記の暗唱」を測る自家汚染になる                 | 4.2.3.2（データ汚染）                     |
| データセット | finding の再発判定が **AI の自由記述タイトルの先頭40文字**頼みで、同じ問題の言い換えを取りこぼす                                    | 8.3.2（重複排除・意味的類似）             |
| データセット | スキル本文が **git 管理されていない**。AI が AI 生成データでスキルを書き換え続けるループに、人間原文のアンカーと差し戻し手段がない   | 8.2.3.3（モデル崩壊・データの出自）       |

個人的に一番痛かったのは分母です。この記事の前半で発生率を全部「概算」と断らざるを得なかった根本原因が、本の指摘（率にできない評価は比較できない）とそのまま一致していました。収集フックはトランスクリプトを既に読んでいるので、フリクションの有無に関係なく「このセッションでどのスキルが使われたか」を1行記録するだけで、以後の発生率は概算でなく実測になります。

### 棚卸し時点の生データ（2026-07-15）

| 項目                                        | 値                                                  |
| ------------------------------------------- | --------------------------------------------------- |
| 未処理のフリクションログ                    | 100 件                                              |
| 処理済み（`processed/`）                    | 125 件                                              |
| 適用記録 `applied.jsonl`                    | 78 レコード（applied 71 / no-op 4 / skipped 2 / deferred 1） |
| うち判定者バージョン付きのレコード          | **0 / 78**                                          |
| 経時比較用ベースラインのスナップショット    | 1 件（比較レポートには 2 件必要）                   |

「判定者バージョン付き 0/78」は面白いデータポイントでした。前節で判定者ドリフト対策として版数ハッシュの自動記録を入れたのですが、既存レコードに遡及しないので、当面はすべて「判定者不明」です。集計コードが「不明」を「不変」と暗黙に混同していたことにも、この数字を出して初めて気づきました。機構を作っても、データが物差しに追いつくまでには運用期間が要る、という現実がそのまま数字に出ています。

## 修正プラン：プロンプトを直す前に、物差しを固定する

棚卸しの結果を修正プランに落とすとき、順序の制約が1つあることに気づきました。**分析器のプロンプトを直すこと自体が「判定者の変更」** なので、先にバージョン記録（物差しの固定）を入れてからでないと、改善したのか判定者が変わっただけなのか、後から区別できなくなります。

そこで4フェーズに分け、プロンプト改訂は最後にまとめて1回で行うことにしました（判定者の世代交代を1度で済ませるためです）。

| フェーズ        | やること                                                                                       | 本の根拠                                     |
| --------------- | ---------------------------------------------------------------------------------------------- | -------------------------------------------- |
| 0. 基盤         | 使用回数の分母記録（発生率を実測に）／スキル群の git 化（人間原文のアンカーと編集履歴）         | 4.3.3.2、8.2.3.3、8.3「元データのコピーを保持」 |
| 1. 物差しの固定 | 分析器のプロンプト＋モデルをハッシュで記録し、モデルはフル ID に固定／再発判定を固定タグに       | 3.4.3.2、8.3.2                               |
| 2. リネージ     | 検証結果と eval 結果を一時ディレクトリでなく永続化／finding 単位で人間の判断を記録（一致率の材料） | 8.2.3.3、3.4.1                               |
| 3. 判定品質     | 分析器プロンプトの一括改訂：few-shot 例・曖昧系の重大度マーカー・eval 汚染ガード                 | 3.4.2（例で一貫性 65%→77.5%）、4.2.3.2       |

few-shot の根拠になっている 3.4.2 の数字は具体的で、評価プロンプトに採点例を含めるだけで GPT-4 の判定一貫性が 65% から 77.5% に上がった、という実験（Zheng et al., 2023）が引かれています。うちの分析器プロンプトは形式の定義だけで実例ゼロだったので、これはそのまま輸入できます。

## 視野を広げる：この仕組みは「SKILL.md のファインチューニングループ」だった

個別の穴より面白かったのは、構造の違いです。この仕組みを本の概念に写像すると、**「SKILL.md を重みとみなしたファインチューニングループ」** になります。

| 本の概念                       | この仕組みでの対応物             |
| ------------------------------ | -------------------------------- |
| モデルの重み（事後学習の対象） | SKILL.md 本文                    |
| 指示・選好データ（8章）        | フリクションログ                 |
| データアノテーター（8.1.4）    | 分析器（headless Claude）        |
| データ検証器（8.2.3.2）        | verifier サブエージェント        |
| 評価ハーネス                   | run_harness                      |
| ベンチマーク                   | ベースラインのスナップショット   |

この写像で見ると、本の世界観との構造差がいくつも浮かびます。

- **評価駆動開発の逆走**: 本は「作る前に評価基準を定義せよ」（4.1）。この仕組みはスキルを基準なしで作り、事後に「失敗」だけを定義している。スキルごとの「成功とは何か」がどこにも書かれていない
- **負例しか集めていない**: 本のデータフライホイール（8.1.4）では正例（望ましい振る舞いの実例）が主役。LIMA の教訓（8.1.1）は「少数の高品質な正例が最も効く」なのに、成功したセッションを全部捨てている
- **厳密に測れるものまで主観評価している**: エラーの大半（exit code、権限拒否）はトランスクリプト内に機械可読で存在するのに、AI の読解で拾っている。本の階層は「機能正確性で測れるものはまずそれで測る」（3.3）
- **事後採掘だけで、実行時の層がない**: 本は開発時評価・本番ガードレール・監視の三層構造（4章と10章）。うちは三層目しかない
- **「削る」キュレーションがない**: 8.1 の定義には「望ましくない振る舞いを忘れさせるためにデータを取り除く」が含まれる。今のループは追記一方向で、陳腐化した記述を検出して削る経路がない

三層の話は、実際の未リカバリログを見るとわかりやすいので、固有の情報を伏せて一部引用します。あるスキルには「処理完了後に定型見出しのレポートを必ず出力する」という指示があるのですが：

> 手順8「出力フォーマット」は（中略）定型見出しを省略せず必ず出力するよう明示的に指示している。しかし本セッションでは（中略）この定型レポートが一度も出力されなかった。（中略）セッション終了まで是正されなかった。 [unrecovered]

「定型見出しが出力されたか」は、本が指示追従ベンチマーク IFEval の文脈で言う「プログラムで自動検証できる指示」（4.1.3.1）そのものです。翌日にログとして採掘して SKILL.md を直すより、セッション中に機械チェックして自己修正させる方が早い。**事後採掘から実行時契約へ**、が構造レベルでの次の一手だと思っています。

逆向きの差もあります。本は「10% の差の検出には約100サンプル」（4.3.3.2）というプロダクション規模の統計を前提にしますが、個人環境の母数は週数十セッションです。この規模で有意性ゲートを真似ると検出力のない儀式になります。ここで輸入すべきは統計ではなく、「まず50件の良質なデータで傾きを見る」（8.1.3）、「データを手で15分見るのが最高の費用対効果」（8.3.1）の方でした。

## 評価は何を軸にするか：北極星・ガードレール・診断の3層

最後に、「じゃあ何を指標にするのか」をもう一段引いて考えます。タスク成功率か、時間か、トークンか。

結論は「1つ選ぶ」ではなく、本の 4.1.4（パレート最適化：妥協できる軸とできない軸をまず分ける）に倣って、**北極星1つ＋妥協不可の閾値＋診断指標**の3層に役割分担させる、です。候補を1つずつ査定すると：

- **タスク成功率**: 北極星の最有力。ただしスキルごとの「成功」定義が前提で、「人間が3回介入して成功」を成功に数えると委任の意味がない
- **時間**: バックグラウンド実行のスキルにはほぼ無意味で、対話スキルには痛い。最適化目標ではなく**閾値**が正しい置き場所（4.1.4 も「高レイテンシーは厄介だが多くの場合致命的ではない」）
- **トークン**: 目標化すると、レポートが薄くなる・検証を飛ばす方向に倒れる。「BLEU を最適化しても機能正確性は上がらない」（3.3.2.2）と同型の代理指標の罠。**成功1回あたりのコスト**という比率で監視に留める
- **ターン数・フリクション数**: 北極星が動いた理由を説明する**診断層**。この記事の前半で測っていたのは、実は全部この層でした

個人の自動化環境で希少な資源は、お金でも計算資源でもなく**自分の注意**です。スキルの存在意義は委任なので、北極星は

**無介入成功率 = 人間の介入なしに、そのスキルの成功条件を満たして完了した割合**

が一番筋が良い、というのが今の結論です。前半で出てきた「未リカバリ率」の一般化になっていて、本の「評価指標をビジネス指標に対応付けろ」（4.3.2.3）もそのまま書けます：無介入 95% なら cron で無人運転してよい、80% なら起動は自動で結果は目視、60% なら対話でしか使わない、という具合です。

| 層                     | 指標                                                     | 役割                                     |
| ---------------------- | -------------------------------------------------------- | ---------------------------------------- |
| 北極星（最大化）       | 無介入成功率（スキル別の成功契約に対して）               | スキルの健康の定義。経時トレンドを追う   |
| ガードレール（閾値）   | 不可逆な事故ゼロ／対話系のみレイテンシー上限             | 下回ったら成功率に関係なく即対処         |
| 診断（説明変数）       | フリクション件数・種別、ターン数、成功1回あたりトークン   | 北極星が動いた理由の特定                 |

やらないと決めたことも2つあります。**合成スコアを作らない**（「成功率 ×0.5 ＋コスト ×0.3…」のような加重平均は、本が公開リーダーボードの均等平均を批判するのと同じ恣意性〔4.2.3.1〕を持ち込む）。**週次のノイズを追わない**（この母数では、月単位のトレンドとガードレール違反の単発検知だけを見る）。

## 結論：効果測定は、直す機構より先に

ここまでを整理すると、skill-fix への打ち手は「評価者（verifier・分析器）自身の質」「測る指標」「設計の外形（実行時契約・正例の収穫）」の3本柱になりました。1本目は原因がはっきりしていたので先に直し、2本目＝指標の付け替え（分母の記録と無介入成功率）がこれからの本丸、3本目は中期の宿題です。

今回いちばんの学びはこれです。

**効果を測る土台を作る前に、直す機構のほうを賢くしようとしていました。**

skill-fix は 71 件の修正を当てていて、「曖昧な指示」は目に見えて減っていました。にもかかわらず、素朴に「エラー発生率」を見ていたせいで「効いていない」ように見えてしまっていました。指標が現実に追いついていなかったわけです。（verifier のバイアスのように原因がはっきりしていて修正も軽いものは別ですが、多くの改善案はそうではありませんでした。）

なので次の一手は、機構の追加ではなく **指標の付け替え** から始めます。承認負荷はゼロで、セクション構造も変えず、「エラー修正が効いているか」を初めて可視化する最小の一歩です。エラーログの分類と扱いを before / after で並べると、狙いがはっきりします。

| 観点             | Before（発生数ベース）                      | After（未リカバリベース）                                                    |
| ---------------- | ------------------------------------------- | ---------------------------------------------------------------------------- |
| エラーごとの記録 | 失敗した事実だけ                            | `[recovered]` / `[unrecovered]` を付け、復帰できたかで区別する               |
| 発生源での分類   | 分けていない                                | 環境／権限／**遵守失敗**（指示に従えなかった失敗）などのタグで分ける（中期） |
| 処理の優先度     | エラーの**発生数**（`error_count`）が多い順 | **未リカバリのエラー**（`unrecovered_count`）を最優先                        |
| 「効いた」の判定 | 発生率が下がったか                          | 未リカバリ率が下がったか（復帰できるようになったか）                         |

いちばん効くのは1行目の `[recovered]` / `[unrecovered]` マーカーです。これが付いて初めて、「発生は止められなくても、ちゃんと復帰できるようになったか」という、いまの修正が実際に効いている軸で測れるようになります。

## まとめ

- 自己改善ループを「作る」段階から「測る」段階に進めてみました
- エラーと曖昧な指示では修正の効き方が真逆で、素朴な発生率では効果が見えないことがわかりました
- verifier 自体も AI なので、冗長性バイアスのような「評価者の癖」を疑う必要がありました
- 『AI Engineering』3・4・8章と突き合わせると、判定者のバージョン管理・分母の欠損・負例偏重など、設計レベルの穴が20個ほど見つかりました
- 評価の軸は1つ選ぶのではなく、北極星（無介入成功率）＋ガードレール＋診断の3層に分けるのが今の結論です
- 教訓は「効果測定は、直す機構より先に作れ」。測れないものは、良くなったかどうかも言えません

前回、この仕組みで目指したいのは **使われるほど自分で良くなっていくドキュメント** だと書きました。ただ「良くなっている」と言い切るには、まず良くなったことを測れないといけない、というのが今回の反省です。

次はまず Phase 0（使用回数の分母記録とスキル群の git 化）から実装して、無介入成功率で測れる状態を作ってから、また続報を書ければと思います。ここまで読んでくださり、ありがとうございました！
