# 導入

みなさんは **`go fix`** コマンドを使ったことがありますか？🤔

`go fix` は Go のコードを自動的に修正してくれる公式ツールです。API の破壊的変更があった際に、古いコードを新しい API に自動変換してくれます。

例えば、Go 1.26 では標準ライブラリにいくつかの変更が加わっていますが、`go fix` を使えば自動的に新しい書き方に変換できます！

## go fix はなぜ高速なのか？

`go fix` コマンドは内部で **静的解析** を行い、コードの抽象構文木（AST）を走査します。

ここで疑問が湧いてきます 💡

「大規模なコードベースでも、なぜ `go fix` は高速に動作するのか？」

従来の AST 走査では、全てのノードを訪問する必要があり、大規模プロジェクトでは時間がかかっていました。しかし、Go のツールチェーンでは **約 2.5 倍の高速化** を実現する仕組みが導入されています。

# ast.Inspect による従来の実装

従来、Go の AST を走査するには **`ast.Inspect`** 関数が使われていました。

`ast.Inspect` は AST を **深さ優先探索（DFS）** で走査します。この実装では、**全てのノードを訪問する必要があり**、計算量は O(N) になります（N はノード数）。

## 従来の実装の課題

特定の型のノード（例えば `*ast.FuncDecl` のみ）をフィルタリングしたい場合でも、**全てのノードを訪問しなければなりません**。

これは大規模なコードベースでは以下のような問題を引き起こします：

- 不要なノードの訪問によるオーバーヘッド
- 型アサーションの繰り返し実行
- 再帰的な関数呼び出しのコスト

この課題を解決するために登場したのが、次に紹介する **`ast.Inspector`** です！

# ast.Inspector による新しい実装

`golang.org/x/tools/go/ast/inspector` パッケージは、従来の `ast.Inspect` の課題を解決する新しいアプローチを提供します。

その核となるのが **`Inspector`** 構造体と **`event`** 構造体です！

## Inspector 構造体

```go
// An Inspector provides methods for inspecting
// (traversing) the syntax trees of a package.
type Inspector struct {
	events []event
}
```

## event 構造体

`Inspector` の中核となるのが `event` 構造体です。

```go
// An event represents a push or a pop
// of an ast.Node during a traversal.
type event struct {
	node   ast.Node
	typ    uint64 // typeOf(node) on push event, or union of typ strictly between push and pop events on pop events
	index  int32  // index of corresponding push or pop event
	parent int32  // index of parent's push node (push nodes only), or packed edge kind/index (pop nodes only)
}
```

各フィールドの役割は以下の通りです：

- **`node`**: AST ノード本体
- **`typ`**: ノードの型情報（ビットマスク形式）
- **`index`**: 対応する push/pop イベントのインデックス
- **`parent`**: 親ノードの情報

## この設計の優れた点

この event ベースの設計には、以下の2つの大きな利点があります 💡

1. **型チェックがビット演算に** - 型アサーションが不要になり、高速な型フィルタリングが可能
2. **サブツリーのスキップが可能** - `typ` フィールドに子孫の型情報が累積されるため、不要なサブツリーの走査を避けられる

## push/pop の実装

AST の走査中に、各ノードに対して **push イベントと pop イベント** が生成されます。

### push の実装

まずは `push` メソッドを見てみましょう。

```go
func (v *visitor) push(ek edge.Kind, eindex int, node ast.Node) {
	var (
		index       = int32(len(v.events))
		parentIndex = v.stack[len(v.stack)-1].index
	)
	v.events = append(v.events, event{
		node:   node,
		parent: parentIndex,
		typ:    typeOf(node), // 型が指定される
		index:  0, // popでセットされる
	})
	v.stack = append(v.stack, item{
		index:            index,
		parentIndex:      parentIndex,
		edgeKindAndIndex: packEdgeKindAndIndex(ek, eindex),
	})

	// 2B nodes ought to be enough for anyone!
	if int32(len(v.events)) < 0 {
		panic("event index exceeded int32")
	}

	// 32M elements in an []ast.Node ought to be enough for anyone!
	if ek2, eindex2 := unpackEdgeKindAndIndex(packEdgeKindAndIndex(ek, eindex)); ek2 != ek || eindex2 != eindex {
		panic("Node slice index exceeded uint25")
	}
}
```

`push` メソッドでは：
- 新しい event を作成し、`events` 配列に追加
- `typ` フィールドに現在のノードの型を設定
- `index` は 0 で初期化（後で `pop` でセットされる）

### pop の実装

```go
func (v *visitor) pop(node ast.Node) {
	top := len(v.stack) - 1
	current := v.stack[top]

	push := &v.events[current.index]
	parent := &v.stack[top-1]

	push.index = int32(len(v.events))              // ここでpushのindexがセットされる
	parent.typAccum |= current.typAccum | push.typ // accumulate type bits into parent

	v.stack = v.stack[:top]

	v.events = append(v.events, event{
		node:   node,
		typ:    current.typAccum,
		index:  current.index,
		parent: current.edgeKindAndIndex, // see [unpackEdgeKindAndIndex]
	})
}
```

`pop` メソッドでは：
- 対応する push event の `index` フィールドに現在の位置をセット
- **親の `typAccum` に子孫の型情報を累積**（これが高速化の鍵！）
- 新しい pop event を `events` 配列に追加

## 重要なポイント

### 1. PUSH/POP のペアリング

各ノードに対して必ず PUSH イベントと POP イベントが生成されます。この構造は「balanced parenthesis tree（バランスのとれた括弧木）」と呼ばれています。

```
PUSH File    → events[0]
  PUSH Ident   → events[1]
  POP Ident    → events[2]
  PUSH FuncDecl → events[3]
    ...
  POP FuncDecl → events[12]
POP File     → events[13]
```

### 2. index フィールドの相互参照

- **PUSH event の `index`**: 対応する POP event のインデックスを指す（`ev.index > i`）
- **POP event の `index`**: 対応する PUSH event のインデックスを指す（`ev.index < i`）

この仕組みにより、以下の操作が O(1) で可能になります：

- あるノードをスキップして次のノードへジャンプ
- サブツリー全体をスキップ

```go
// Preorder の実装より
if ev.index > i {
    // push event
    pop := ev.index
    if in.events[pop].typ&mask == 0 {
        // サブツリーに対象の型がない場合、popへジャンプしてスキップ
        i = pop + 1
        continue
    }
}
```

### 3. typ フィールドの累積（typAccum）

POP event の `typ` フィールドには、そのノードの**子孫に含まれる全ての型のビットマスク**が累積されます：

```go
// pop の実装より
parent.typAccum |= current.typAccum | push.typ
```

これにより、サブツリーを探索する前に「このサブツリーに目的の型が含まれているか」を O(1) で判定できます。

例えば、`*ast.FuncDecl` の POP event (Step 12) の typ には：

- `typeOf(*ast.Ident)` (関数名)
- `typeOf(*ast.FuncType)` (関数型)
- `typeOf(*ast.FieldList)` (パラメータリスト)
- `typeOf(*ast.BlockStmt)` (関数本体)

これらの OR が格納されます。

### 4. parent フィールドの二重用途

`parent` フィールドは PUSH と POP で異なる意味を持ちます：

- **PUSH event**: 親ノードの PUSH event のインデックス
- **POP event**: edge kind と index がパックされた値

```go
// PUSH 時
parent: parentIndex

// POP 時
parent: packEdgeKindAndIndex(edge.Kind, index)
```

これにより、親子関係とエッジの種類（どのフィールドからの参照か）の両方を保持できます。

## なぜこの実装が高速なのか

1. **型フィルタリングがビット演算**: `typ & mask` による O(1) の型チェック
2. **サブツリースキップ**: 対象の型を含まないサブツリーを O(1) でスキップ
3. **動的ディスパッチの削減**: `ast.Inspect` のような再帰的な関数呼び出しやインターフェース呼び出しが不要

従来の `ast.Inspect` では全ノードを訪問する必要がありましたが、`inspector.Inspector` では事前に構築した events 配列を線形走査し、不要なサブツリーをスキップできるため、約 2.5 倍高速になっています。

最終的な `v.events`　の形式

```go
v.events = []xxx{
    // Index 0: File PUSH
    {
      node:   *ast.File,
      parent: -1,
      typ:    typeOf(*ast.File),
      index:  13,  // 対応するPOPのインデックス
    },

    // Index 1: File.Name (Ident) PUSH
    {
      node:   *ast.Ident{Name: "main"},
      parent: 0,
      typ:    typeOf(*ast.Ident),
      index:  2,
    },

    // Index 2: File.Name (Ident) POP
    {
      node:   *ast.Ident{Name: "main"},
      typ:    0,  // 子がないのでtypAccumは0
      index:  1,  // 対応するPUSHのインデックス
      parent: packEdgeKindAndIndex(edge.File_Name, -1),
    },

    // Index 3: FuncDecl PUSH
    {
      node:   *ast.FuncDecl,
      parent: 0,
      typ:    typeOf(*ast.FuncDecl),
      index:  12,
    },

    // Index 4: FuncDecl.Name (Ident) PUSH
    {
      node:   *ast.Ident{Name: "main"},
      parent: 3,
      typ:    typeOf(*ast.Ident),
      index:  5,
    },

    // Index 5: FuncDecl.Name (Ident) POP
    {
      node:   *ast.Ident{Name: "main"},
      typ:    0,
      index:  4,
      parent: packEdgeKindAndIndex(edge.FuncDecl_Name, -1),
    },

    // Index 6: FuncType PUSH
    {
      node:   *ast.FuncType,
      parent: 3,
      typ:    typeOf(*ast.FuncType),
      index:  9,
    },

    // Index 7: FieldList (Params) PUSH
    {
      node:   *ast.FieldList,
      parent: 6,
      typ:    typeOf(*ast.FieldList),
      index:  8,
    },

    // Index 8: FieldList (Params) POP
    {
      node:   *ast.FieldList,
      typ:    0,  // List が空なので0
      index:  7,
      parent: packEdgeKindAndIndex(edge.FuncType_Params, -1),
    },

    // Index 9: FuncType POP
    {
      node:   *ast.FuncType,
      typ:    typeOf(*ast.FieldList),  // 子から累積
      index:  6,
      parent: packEdgeKindAndIndex(edge.FuncDecl_Type, -1),
    },

    // Index 10: BlockStmt PUSH
    {
      node:   *ast.BlockStmt,
      parent: 3,
      typ:    typeOf(*ast.BlockStmt),
      index:  11,
    },

    // Index 11: BlockStmt POP
    {
      node:   *ast.BlockStmt,
      typ:    0,  // List が空なので0
      index:  10,
      parent: packEdgeKindAndIndex(edge.FuncDecl_Body, -1),
    },

    // Index 12: FuncDecl POP
    {
      node:   *ast.FuncDecl,
      typ:    typeOf(*ast.Ident) | typeOf(*ast.FuncType) | typeOf(*ast.FieldList) | typeOf(*ast.BlockStmt),
      index:  3,
      parent: packEdgeKindAndIndex(edge.File_Decls, 0),
    },

    // Index 13: File POP
    {
      node:   *ast.File,
      typ:    typeOf(*ast.Ident) | typeOf(*ast.FuncDecl) | typeOf(*ast.FuncType) | ...,
      index:  0,
      parent: packEdgeKindAndIndex(edge.Invalid, -1),
    },
}

```

これで `events` 配列には、AST の全ノードの push/pop イベントが balanced parenthesis tree 構造で格納されます！

# 実際のコード例で動作を確認してみる！

ここまでの説明で、`ast.Inspector` の内部実装が理解できたと思います。

では、実際に従来の `ast.Inspect` と新しい `ast.Inspector` を使ったコードを比較してみましょう！

## 解析対象のコード

以下のようなシンプルな Go コードから、関数宣言を見つけるプログラムを考えます：

```go
package main

func hello() {
    println("Hello, World!")
}

func goodbye() {
    println("Goodbye!")
}
```

## 従来の ast.Inspect を使った実装

```go
package main

import (
    "go/ast"
    "go/parser"
    "go/token"
    "fmt"
)

func findFunctionsWithInspect(src string) {
    fset := token.NewFileSet()
    file, _ := parser.ParseFile(fset, "", src, 0)

    // 全てのノードを訪問する
    ast.Inspect(file, func(n ast.Node) bool {
        // 型アサーションで FuncDecl をチェック
        if fn, ok := n.(*ast.FuncDecl); ok {
            fmt.Println("Found:", fn.Name.Name)
        }
        return true // 全てのノードを訪問し続ける
    })
}
```

このコードでは、**全てのノード**を訪問し、毎回型アサーションを行っています。

## 新しい ast/inspector.Inspector を使った実装

```go
package main

import (
    "go/ast"
    "go/parser"
    "go/token"
    "fmt"
    "golang.org/x/tools/go/ast/inspector"
)

func findFunctionsWithInspector(src string) {
    fset := token.NewFileSet()
    file, _ := parser.ParseFile(fset, "", src, 0)

    // Inspector を作成
    inspect := inspector.New([]*ast.File{file})

    // フィルタで FuncDecl のみを指定
    nodeFilter := []ast.Node{
        (*ast.FuncDecl)(nil),
    }

    // Preorder で走査（フィルタに一致するノードのみ訪問）
    inspect.Preorder(nodeFilter, func(n ast.Node) {
        fn := n.(*ast.FuncDecl)
        fmt.Println("Found:", fn.Name.Name)
    })
}
```

## 2つの実装の違い

新しい `ast.Inspector` の実装では：

1. **`nodeFilter` で型を指定** - 事前に欲しい型を宣言
2. **型アサーションが不要** - コールバック関数内では既に型が確定
3. **不要なサブツリーをスキップ** - 前述の `typ` フィールドの累積により、`*ast.FuncDecl` を含まないサブツリーは走査されない

この仕組みにより、**大規模なコードベースでも高速に特定のノードを見つけられる**わけです！

## go fix での実際の使われ方

`go fix` コマンドも内部で `ast.Inspector` を使用しています。

例えば、特定の関数呼び出しを新しい API に置き換える際：
1. `*ast.CallExpr` ノードのみをフィルタリング
2. 関数名をチェック
3. 必要に応じて AST を書き換え

このように、必要なノードのみを効率的に見つけることで、大規模プロジェクトでも高速に動作します 🚀
