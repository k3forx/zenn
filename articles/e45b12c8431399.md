## 導入

Go公式ブログの [Type Construction and Cycle Detection](https://go.dev/blog/type-construction-and-cycle-detection) を読んでいたら、以下の記述が目に留まりました。

> Go 1.26 significantly improved the type checker's handling of type construction and cycle detection.

※意訳: Go1.26では、型チェッカーにおける型コンストラクションとサイクル検出の扱いが大幅に改善された。

普段Goを書いていて「型のサイクル」を意識することはほとんどないのですが、記事の中には `type T [unsafe.Sizeof(T{})]int` のような見慣れない書き方が出てきます。これは配列のサイズを自分自身 (`T{}`) のサイズで決めようとしていて、型の構築が自己参照してしまう例です。「これが型レベルでのサイクルか〜」と興味を惹かれました。

このブログではGo1.26で入った型コンストラクション・サイクル検出の改善を、公式ブログを追いながら自分なりに噛み砕いてみたいと思います。

このブログ内で使用するコードはGo1.26で書かれています。ブログ執筆時点で手元にGo1.26を用意するのが難しいケースもあると思うので、コードは動かせないものも含まれる点をご了承ください。

## この記事で知れること

- Goの型チェッカーがどうやって型を組み立てるか (型コンストラクション)
- 再帰的な型がなぜ許されて、どこから許されなくなるのか
- サイクル検出の実装方針 (upstream / downstream という考え方)

## そもそも型チェッカーは何をしているのか

[公式ブログ](https://go.dev/blog/type-construction-and-cycle-detection) の冒頭にわかりやすい説明があります。

> Go's static typing eliminates whole classes of errors at compile time. When a Go package is compiled, the source code is converted into an abstract syntax tree (AST), which is then passed to the Go type checker.

※意訳: Goの静的型付けは、コンパイル時にエラーの大部分を除去する。Goのパッケージをコンパイルするとき、ソースコードは抽象構文木 (AST) に変換され、Goの型チェッカーに渡される。

型チェッカーの仕事は大きく2つに分けられます。

1. ASTに現れる型が妥当かを確かめる (例: mapのキーは `comparable` でなければならない)
2. その型を使った操作が妥当かを確かめる (例: `int` と `string` を足し算してはいけない)

**この記事で扱うのは「1. 型が妥当かを確かめる」のための内部処理である「型コンストラクション」と、そこで起こる可能性のあるサイクルの検出**になります。

## 型コンストラクションを追ってみる

### まずはシンプルな例から

以下のようなコードを考えます。

```go
type T []U
type U *int
```

型チェッカーは、定義された型 (Defined type) ごとに内部的な `Defined` という構造体を作るとのことです。それぞれの `Defined` は型式 (type expression) に対応する型を指すポインタを持っていて、今回の例だと以下のような関係になります。

- `T` の `Defined` は `Slice` 構造体を指す
- その `Slice` の要素型は `U` の `Defined` を指す
- `U` の `Defined` は `Pointer` 構造体を指す
- その `Pointer` は組み込み型の `int` を指す

ここで公式ブログに重要な記述があります。

> Type construction is a depth-first process: a type becomes complete (all fields populated and dependencies complete) only after its dependencies are complete.

※意訳: 型コンストラクションは深さ優先で進む。型がcomplete (すべてのフィールドが埋まり、依存する型もすべてcomplete) になるのは、その依存がすべてcompleteになった後である。

つまり **ある型がcompleteになるには、その型が依存しているすべての型が先にcompleteになっている必要がある** ということですね。先ほどの例で言うと、`int` → `*int` (`U`) → `[]U` (`T`) の順でcompleteになるイメージです。

### 再帰的な型はどう扱うのか

「依存がすべてcompleteになってから自分もcompleteになる」というルールを厳密に適用すると、再帰的な型は作れなくなってしまいます。

たとえば以下のような型は実際にGoで書けるので、なんらかの仕組みで許容されているはずです。

```go
type T []U
type U *T
```

公式ブログでは以下のように説明されていました。

> In recursive types, the type checker can point to an incomplete type. When evaluating \*T, the base type simply points to T even though T is still under construction. When T finishes construction, the "loop" of types closes and all types in the loop complete simultaneously.

※意訳: 再帰的な型の場合、型チェッカーはincompleteな型を指すことが許される。`*T` を評価するとき、たとえ `T` がまだ構築中でも、ベース型はただ `T` を指すだけである。`T` の構築が終わると型の「ループ」が閉じ、そのループに含まれるすべての型が同時にcompleteになる。

**型のポインタは、その指す先がincompleteでも持てる**、ということが書かれています。依存先がcompleteになる必要があるのは「型そのものの構築」ではなく「型を分解する (deconstruct) 操作」のほう、というのがキモのようです。

公式ブログの表現が端的なので、もう一度引用しておきます。

> Type completeness does not block type construction itself, only certain operations that deconstruct types.

※意訳: 型のcompleteness は、型コンストラクションそれ自体をブロックするのではなく、型を分解するような一部の操作だけをブロックする。

### 再帰 + 値になると話が変わる

ここまでは「再帰的でも型はなんとか作れる」という話でした。では型チェッカーが壊れるのはどんなときでしょうか？

公式ブログでは以下のコードが例として挙げられています。

```go
type T [unsafe.Sizeof(T{})]int
```

見慣れない形ですが、これは「要素数が `unsafe.Sizeof(T{})` で決まる `int` 配列を `T` として定義する」というコードです。完成形では以下のループができあがってしまいます。

- `T` がcompleteになるには配列の要素数、つまり `T{}` のサイズを知る必要がある
- `T{}` のサイズを知るには `T` の構造をdeconstructする必要がある
- しかし `T` はまだ構築中なのでdeconstructできない

**依存が相互にdeconstructを要求しあっていて、どちらも先に進めない**状態になります。これは先ほどの `type T []U; type U *T` のようにポインタで間接的に参照しているケースとは違って、型を「使う」時点で中身を覗かないといけない形になっています。

このような型はコンパイル時に検出してエラーにするしかなく、それを担当しているのが今回のテーマである **サイクル検出** です。

## サイクル検出はどこで走るのか

### incompleteな値

サイクル検出を考える上で、公式ブログでは「incompleteな値 (incomplete value)」という概念が導入されます。

> Incomplete values are those whose type is incomplete. For instance, `T{}` is an incomplete value of type `T`.

※意訳: incompleteな値とは、その型がincompleteである値のこと。たとえば `T{}` は型 `T` のincompleteな値である。

incompleteな値を使った操作は、その操作が型をdeconstructしない限りは問題ありません。公式ブログで挙げられている例を並べると以下のようになります。

```go
// soundな例 (どんな型でもポインタのサイズは決まっているのでdeconstruct不要)
type T [unsafe.Sizeof(new(T))]int

// unsoundな例 (配列サイズを知るためにTをdeconstructする必要がある)
type T [unsafe.Sizeof(T{})]int
```

:::message
incompleteな値の型が「defined type」そのものになる場合は、その時点で常にunsoundとのこと。定義された型名だけでは内部構造が分からないため、結局どこかでdeconstructが必要になってしまうからのようです。
:::

### upstream / downstream という整理

incompleteな値がどこから流れてきて、どこで問題になるかを整理するために、公式ブログでは **upstream** と **downstream** という言葉が使われています。

- upstream: incompleteな値を**生み出す**式 (型変換、関数呼び出し、型アサーションなど)
- downstream: incompleteな値を**消費する**演算子 (indexing、slicing、関数呼び出しなど)

upstreamの具体例は以下のとおりです。いずれも結果の型がincompleteな `T` になる可能性がある式になっています。

```go
type T [unsafe.Sizeof(T(42))]int                // 型変換

func f() T
type T [unsafe.Sizeof(f())]int                  // 関数呼び出し

var i interface{}
type T [unsafe.Sizeof(i.(T))]int                // 型アサーション

type T [unsafe.Sizeof(<-(make(<-chan T)))]int   // チャネル受信

type T [unsafe.Sizeof(make(map[int]T)[42])]int  // マップアクセス

type T [unsafe.Sizeof(*new(T))]int              // ポインタのデリファレンス
```

### 実装方針: upstreamで食い止める

公式ブログを読む限り、Go1.26のサイクル検出はこんな方針のようです。

- incompleteな値はupstreamで必ず生成される
- なので **upstreamが評価されるタイミングでcompletenessをチェック**すればよい
- incompleteな型を使おうとしていたらエラーを報告し、以後の処理にはinvalidな値を流す

公式ブログに載っている実装イメージは以下のようになっています。

```go
func callExpr(call *syntax.CallExpr) operand {
  x := typeOrValue(call.Fun)
  switch x.mode() {
  // ... other cases
  case typeExpr:
    // T(), meaning it's a conversion
    T := x.typ()
    if !isComplete(T) {
      reportCycleErr(T)
      return invalid
    }
    // ... handle the conversion, T *is* safe to deconstruct
  }
}
```

`T(...)` のような型変換を処理するcase文の中で、型 `T` がcompleteかどうかを `isComplete(T)` でチェックし、incompleteなら `reportCycleErr(T)` でエラーを出して `invalid` を返すという流れです。

**downstream側 (indexingや関数呼び出しなど) で都度チェックするのではなく、upstream側でincompleteな値を外に漏らさないようにする**、というのがシンプルで良さそうだなと感じました。

## 旧実装と何が違ったのか

ここまで読むと「以前はどうやっていたのだろう？」というのが気になってきます。公式ブログには以下のように書かれていました。

> The previous approach used a more complex type construction algorithm with bespoke cycle detection that didn't always work. The new, simpler approach fixed numerous compiler panics (issues #75918, #76383, #76384, #76478, and more), resulting in a more stable compiler.

※意訳: 以前のアプローチはより複雑な型コンストラクションアルゴリズムと、場所ごとに特化したサイクル検出を使っていたが、常にうまくいくとは限らなかった。より単純な新しいアプローチは、多くのコンパイラのパニック (issues #75918, #76383, #76384, #76478 など) を修正し、より安定したコンパイラにつながった。

**「場当たり的なサイクル検出ロジック」を「upstreamで一律にチェックする」方針に整理したことで、コンパイラのパニックがまとめて消えた**というのは面白いストーリーだなと思いました。複雑なコードをシンプルなルールで置き換えると、今まで見えていなかったバグまで一緒に消える、みたいな話は他の領域でもよく聞く気がします。

## まとめ

今回はGo1.26で入った型コンストラクションとサイクル検出の改善を追ってみました。軽くまとめると以下のような内容でした。

- 型コンストラクションは深さ優先で進み、依存がcompleteになってから自分もcompleteになる
- ただしポインタなどによる再帰的な型は、incompleteな型を指したまま構築を続けられる
- 型を分解する (deconstructする) 操作だけは、相手がcompleteになるまで許されない
- サイクル検出はincompleteな値の発生源 (upstream) で一括チェックするように整理された

普段コードを書くだけだとなかなか意識しない層ですが、`type T [unsafe.Sizeof(T{})]int` のような一見奇妙なコードがどう扱われるかを知れて楽しかったです。より詳細を知りたい方は [公式ブログ](https://go.dev/blog/type-construction-and-cycle-detection) を覗いてみるとよいと思います！
