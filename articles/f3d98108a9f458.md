## 前提のおさらい: スタック割り当てとヒープ割り当ては何が違うのか

まずはスタックとヒープの違いを軽く整理しておきます。Goが管理するメモリ領域は、大きく分けるとこの2つに分類されます。

**スタック割り当て**は以下のような性質を持ちます。

- 関数の終了とともにメモリが自動で解放されるため、GC (ガベージコレクタ) の追跡対象にならない
- 関数呼び出しのたびに領域が再利用される
- スタックフレーム内のローカル変数が連続したメモリ領域に配置されるので、CPUキャッシュに乗りやすい

一方、**ヒープ割り当て** は以下の性質を持ちます。

- 割り当てられたメモリはGCが管理し、不要になればGCが回収する
- GCの実行頻度や負荷が増えると、アプリケーション全体のパフォーマンスに影響する

---

Goのコンパイラは開発者が書いた `new(T)` や `make([]T, ...)` のようなコードを、必ずヒープに置くわけではありません。**エスケープ解析 (escape analysis)** を行い、その変数が関数の外に「逃げない」と判断できれば、スタックに置いてくれることがあります。

「エスケープする」というのは、変数への参照が関数の外 (呼び出し元や別のゴルーチン、ヒープ上の構造体の中など) に渡ってしまい、関数終了後もメモリが生き残っている必要がある、という意味です。逃げてしまうと関数終了時に解放してはいけないので、ヒープに置くしかありません。

今回の最適化は、エスケープしないスライスをより広くスタックに載せられるようにするだけでなく、最終的にエスケープする場合でも途中段階をスタックで処理してヒープ割り当てを減らす方向に進んでいます。

## 出発点の問題: スライスの「スタートアップフェーズ」のオーバーヘッド

まずは以下のようなコードから考えてみます。

```go
func process(c chan task) {
	var tasks []task
	for t := range c {
		tasks = append(tasks, t)
	}
	processAll(tasks)
}
```

チャネルから流れてくる `task` を受け取って、すべて受け取り終わったら `processAll` に渡す、という処理です。

しかし、このコードは内部的には以下のような動きをしています。

1. 最初の `append` で容量1のバッキングストアがヒープに割り当てられる
2. 2回目の `append` で容量が足りなくなり、容量2の新しいバッキングストアがヒープに割り当てられる。古い容量1のバッキングストアはガベージになる
3. 3回目の `append` で容量4のバッキングストアが新たに割り当てられ、容量2の方はガベージになる
4. 以降、8、16、32... と倍々に拡張されていく

記事ではこの中間のアロケーションの連なりを「**スタートアップフェーズ (startup phase)**」と呼んでいて、最終的な容量にたどり着くまでに発生する一連の無駄なヒープアロケーション、と説明されています。

ここからは既存の最適化1つと、Go 1.25・1.26で新たに導入された3つの最適化を、「スタートアップフェーズのオーバーヘッドをどうやって削るか」というテーマに沿って順番に見ていきます。

## Go 1.24以前: 容量が定数の `make` をスタックに置く

以前から入っている最適化は、**「容量が定数の `make` で作られたスライスのバッキングストアを、エスケープしなければスタックに置く」** というものです。

つまり、開発者が「おそらく最大10個くらいだろう」と見積もって、以下のように書いたコードが対象になります。

```go
func process2(c chan task) {
	tasks := make([]task, 0, 10) // おそらく最大10個のタスク
	for t := range c {
		tasks = append(tasks, t)
	}
	processAll(tasks)
}
```

コンパイラは以下の2つの条件を満たしたときにバッキングストアをスタックに置きます。

1. **容量が定数である** (この例では10)
2. **バッキングストアが `processAll` 内部からヒープにエスケープしない** (エスケープ解析で確認される)

この2つが満たされれば、`process2` 関数内でのヒープアロケーションは0回になります。もちろん、容量10を超えて `append` してしまうとバッキングストアの拡張が必要になり、その時点で新しいバッキングストアがヒープに割り当てられます。

:::message
この最適化の対象はあくまで「容量が**定数**のとき」です。たとえば引数で渡ってきた変数を容量に指定した場合は対象外です。その制約を取り払いにいくのが次のGo 1.25になります。
:::

### 実際にベンチマークを計測してみた

計測対象のコードは以下です (`//go:noinline` を付けているのはインライン展開による最適化差分を避けるためです)。

```go
//go:noinline
func Scenario1ConstantMake() int {
	s := make([]int, 0, 10)
	for i := range 10 {
		s = append(s, i)
	}
	total := 0
	for _, v := range s {
		total += v
	}
	return total
}
```

| Goのバージョン | ns/op | B/op | allocs/op |
| -------------- | ----: | ---: | --------: |
| 1.24 |  9.18 |    0 |     **0** |
| 1.25 |  9.27 |    0 |     **0** |
| 1.26 |  9.12 |    0 |     **0** |

この例では Go 1.24 の時点で既に 0 allocs で、ns/op もほぼ横ばいです。Go 1.24 の最適化の本領は「他の関数を経由するなど、より複雑なパターンにもスタック割り当てが適用できるようになった」点にあり、この単純なケースの表からは読み取れません。

## Go 1.25: スライスアロケーション箇所に32バイトのスタックバッファを埋め込む

Go 1.25 で入った最適化は、コンパイラが **「スライスを確保する箇所 (スライスアロケーション箇所)」** に対して、あらかじめ32バイトのスタックバッファを埋め込んでおく、というものです。対象になるアロケーション箇所は以下の2パターンあり、**どちらも裏で効いている仕組みは同じ** です。

- `make([]T, 0, n)` で **変数** `n` を容量に指定する呼び出し
- `var s []T` で宣言されたあとの **最初の `append`** (スライスが nil で `len = cap = 0` の状態から拡張される箇所)

まずは `make` 版から見ていきます。容量を引数で指定できるようにした、以下のコードを考えます。

```go
func process3(c chan task, lengthGuess int) {
	tasks := make([]task, 0, lengthGuess)
	for t := range c {
		tasks = append(tasks, t)
	}
	processAll(tasks)
}
```

`lengthGuess` は実行時にしかわからないので、これまでの最適化は使えません。ここで「じゃあ実行時に動的にスタックフレームのサイズを変えればいいのでは？」と思いたくなりますが、記事中には以下のような一節があります。

> Go stacks do not have any `alloca`-style mechanism for dynamically-sized stack frames. All Go stack frames are constant sized.

(意訳)
Goのスタックには、Cの `alloca` のような動的サイズのスタックフレームを扱う仕組みは存在しない。すべてのGoのスタックフレームは固定サイズである。

**Cのように動的にスタックフレームを伸ばすことはできない**、というのが大前提なんですね。

---

これを踏まえてGo 1.25のコンパイラがとった方針は、**「コンパイル時に32バイトの小さなスタックバッファをスタックフレームに埋め込んでおき、実行時に要求サイズに応じて使い分ける」** というものです。

大まかな流れは以下になります。

1. コンパイラが `make` または `append` の呼び出し箇所に、32バイトのスタックバッファを事前に確保するコードを差し込む
1. 実行時に要求サイズ (`要素サイズ × 要求容量`) が32バイトに収まればスタックバッファを使う
1. 32バイトに収まらない場合は、従来通りヒープにフォールバックする

:::message
「32バイトに収まる」とは文字通りバイト数で判定されるので、たとえば要素が `int64` (8バイト) のスライスなら長さ4までしかスタックバッファで賄えません。要素が `byte` (1バイト) なら長さ32までスタックに載ります。
:::

ポイントは、**スタックフレームは依然として固定サイズのまま**だという点です。32バイトという小さな固定サイズのバッファをあらかじめ埋め込んでおくことで、「動的にスタックフレームを伸ばせない」という制約を回避しています。

::: alert
公式ブログは後半で「In Go 1.26, we allocate the same kind of small, speculative backing store on the stack, but now we can use it **directly at the append site**.」と述べ、`append` 箇所へのスタックバッファ埋め込みを **Go 1.26 の機能** として紹介しています。しかし手元の計測では、**非エスケープの `append` については Go 1.25 の時点で既に 0 allocs になっており**、エスケープ解析にも `append does not escape` が 1.25 から登場します。Go 1.26 が本当に新しく加えているのは、次のセクションで紹介する「**エスケープするケース** を救う `runtime.move2heap()`」のほうだと解釈しましたが、間違いがあれば教えていただければと思います
:::

### 計測 1: 変数容量の `make`

計測対象は、容量 `n` を引数で受け取る以下のコードです。

```go
//go:noinline
func Scenario2VariableMake(n int) int {
	s := make([]int, 0, n)
	for i := range n {
		s = append(s, i)
	}
	total := 0
	for _, v := range s {
		total += v
	}
	return total
}
```

`int` は 8 バイトなので、`n = 3` なら `8 × 3 = 24B` で32バイトバッファに収まり、`n = 100` なら 800B で収まりません。この境界の両側で挙動が変わるはずです。

**Small (n=3, 24 bytes ≤ 32B バッファ)**

| Goのバージョン | ns/op | B/op | allocs/op |
| -------------- | ----: | ---: | --------: |
| 1.24 | 15.97 |   24 |         1 |
| 1.25 |  5.15 |    0 |     **0** |
| 1.26 |  5.38 |    0 |     **0** |

**Large (n=100, 800 bytes > 32B バッファ)**

| Goのバージョン | ns/op | B/op | allocs/op |
| -------------- | ----: | ---: | --------: |
| 1.24 | 142.0 |  896 |         1 |
| 1.25 | 146.0 |  896 |         1 |
| 1.26 | 161.9 |  896 |         1 |

Small は Go 1.25 で `1 alloc → 0 allocs`、実行時間も約3倍高速化しています。`int` 1要素は8バイトなので、`n = 3` のとき `8 × 3 = 24` バイトとなり、32バイトのスタックバッファに収まるおかげです。

一方 Large (800B > 32B) は全バージョンで 1 alloc のまま。**「32バイトバッファに収まらなければ従来通りヒープにフォールバック」** という挙動が明確に読み取れます。

### 計測 2: `var` + `append`

もう1つのアロケーション箇所は、冒頭のスタートアップフェーズの例 (`process` 関数) のように、`var tasks []task` とだけ宣言して `append` で積んでいく書き方です。`make` を一切書かないので「どこにスタックバッファを埋め込むのか？」と感じそうですが、**コンパイラは `append` の呼び出し箇所自体をスタックバッファ埋め込みの対象にしています**。動作イメージは以下の通りです。

1. 最初の数回の `append` では、スタック上のバッファにデータが書き込まれる
2. スタックバッファが満杯になったら、それ以降はヒープに大きめの容量を確保してそちらを使う
3. スタックバッファに収まる程度の小さなスライスであれば、ヒープアロケーションは一切発生しない

これで、冒頭の `process` のような「一切 `make` を書いていないコード」でも、**開発者がコードを書き換えなくてもスタートアップフェーズのオーバーヘッドが削減される**、という嬉しい状態になります！

計測対象は、`make` を使わず `var` だけで宣言して `append` で積み上げるコードです。

```go
//go:noinline
func Scenario3AppendOnly(n int) int {
	var s []int
	for i := range n {
		s = append(s, i)
	}
	total := 0
	for _, v := range s {
		total += v
	}
	return total
}
```

**Small (n=3)**

| Goのバージョン | ns/op | B/op | allocs/op |
| -------------- | ----: | ---: | --------: |
| 1.24 | 41.19 |   56 |         3 |
| 1.25 |  5.11 |    0 |     **0** |
| 1.26 |  5.40 |    0 |     **0** |

**Large (n=100)**

| Goのバージョン | ns/op | B/op | allocs/op |
| -------------- | ----: | ---: | --------: |
| 1.24 | 285.4 | 2040 |         8 |
| 1.25 | 244.2 | 1984 |     **5** |
| 1.26 | 240.5 | 1984 |         5 |

Small は Go 1.25 で 3 allocs → 0 allocs、約8倍高速化しています。Large も Go 1.25 で 8 → 5 allocs に減っています 🎉

## Go 1.26: エスケープする `append` を `runtime.move2heap()` で救う

ここまでの最適化には共通の前提条件がありました。それは **「バッキングストアが関数の外にエスケープしないこと」** です。

でも、関数の戻り値としてスライスを返すケースはどうでしょうか？

```go
func extract(c chan task) []task {
	var tasks []task
	for t := range c {
		tasks = append(tasks, t)
	}
	return tasks
}
```

この場合、戻り値として外に持ち出される以上、バッキングストアは必ずヒープ上になければいけません。関数が終了しても生き残っていないといけないからです。これまでの話だと、ここはまだ最適化の対象外です。

---

従来、このケースを手で最適化しようとすると、以下のように **最終サイズがわかった時点で1回だけ `make` + `copy` する** パターンがよく使われていました。

```go
func extract2(c chan task) []task {
	var tasks []task
	for t := range c {
		tasks = append(tasks, t)
	}
	tasks2 := make([]task, len(tasks))
	copy(tasks2, tasks)
	return tasks2
}
```

でも、このパターンには地味に欠点があります。**`tasks` がたとえ小さくてスタックバッファに収まるサイズであっても、最後の `make` + `copy` は必ず実行されてしまう**ことです。その分、余計なヒープアロケーションとコピーが発生してしまうわけですね。

---

そもそも「なぜ `make` + `copy` が最適化になるのか？」というと、**エスケープ解析の対象を `tasks` と `tasks2` に分けている**のがポイントです。

素朴な `extract` では `tasks` 自体が戻り値として外に逃げるので、コンパイラは append の途中経過のバッキングストアも含めて全部ヒープに置くしかありません。スタートアップフェーズで cap 1 → 2 → 4 → 8 → 16 と拡張される過程のガベージがすべてヒープ上に積み上がってしまいます。

一方 `extract2` では、`tasks` は関数内で `copy` の読み取り元にしか使われず、戻り値にもなりません。**エスケープするのは `tasks2` だけ**なので、コンパイラは「`tasks` のバッキングストアは関数内で閉じる」と判断できます。その結果、

1. append の途中経過 (スタートアップフェーズの中間バッキングストア) が Go 1.26以降であればスタック上の32バイトバッファで賄える
2. 最後の `make([]task, len(tasks))` で **ちょうど `len` ぴったりのサイズ** を1回だけヒープ確保する (倍々で膨らんだ余分な capacity を切り詰められる)
3. `copy` でスタック → ヒープにデータを移して返す

という流れになり、「**エスケープする範囲を最終結果だけに絞ることで、途中経過をスタックに押し込める**」手動最適化として成立していたわけです。

---

Go 1.26のコンパイラは、これを別のアプローチで自動化するようになっています。具体的には、**戻り値の直前に `runtime.move2heap()` の呼び出しをコンパイラが自動で挿入する**、と公式ブログで説明されています。

`runtime.move2heap()` は公式ブログで _"a special compiler+runtime function"_ と表現されている、通常のGo関数とは異なる関数です。`runtime` パッケージから呼び出せる公開 API ではなく、コンパイラとランタイムが協調して実現する擬似関数であるという点に注意してください。

具体的には、以下のように `return` の手前に `runtime.move2heap()` の呼び出しが自動で挿入されていると考えてください。

```go
func extract3(c chan task) []task {
	var tasks []task
	for t := range c {
		tasks = append(tasks, t)
	}
	tasks = runtime.move2heap(tasks) // コンパイラが同等の処理を自動で挿入
	return tasks
}
```

差し込まれる `runtime.move2heap()` は以下のように動作します。

1. **渡されたスライスがすでにヒープ上にある場合**: 何もせずそのまま返す (恒等関数として動作)
2. **スライスがスタック上にある場合**: 正確なサイズのヒープ領域を1回だけ割り当て、データをコピーしてヒープ上のスライスを返す

**`extract2` の手動パターンとの違い**は、「スライスがすでにヒープにある場合には何もしない」という点です。

- スタックバッファに収まるケース: ループ中はスタック上で積まれ、最後に1回だけヒープに正確なサイズでコピーされる
- スタックバッファに収まらず、途中でヒープにあふれたケース: 最後の `runtime.move2heap()` は何もせず、既存のヒープバッキングストアがそのまま使われる

:::message
`extract2` と `extract3` は似ているように見えますが、「バッキングストアがすでにヒープにある場合に、余分な `make` + `copy` が発生するかどうか」という点で挙動が違います。コンパイラによる自動挿入の方が賢い動きをしてくれる、と捉えるとわかりやすいかなと思います。
:::

### 実際にベンチマークを計測してみた

計測対象は Scenario 3 とほぼ同じで、最後に `return s` でスライスを返す (= エスケープする) ようにしたコードです。

```go
//go:noinline
func Scenario4AppendAndReturn(n int) []int {
	var s []int
	for i := range n {
		s = append(s, i)
	}
	return s
}
```

**Small (n=3)**

| Goのバージョン | ns/op | B/op | allocs/op |
| -------------- | ----: | ---: | --------: |
| 1.24 | 41.61 |   56 |         3 |
| 1.25 | 44.62 |   56 |         3 |
| 1.26 | 12.97 |   24 |     **1** |

**Large (n=100)**

| Goのバージョン | ns/op | B/op | allocs/op |
| -------------- | ----: | ---: | --------: |
| 1.24 | 288.8 | 2040 |         8 |
| 1.25 | 282.7 | 2040 |         8 |
| 1.26 | 224.5 | 1984 |     **5** |

エスケープするスライスに対しては Go 1.25 までスタック最適化が一切効かず、Go 1.26 で一気に状況が変わります。Small は 3 allocs → 1 alloc、Large は 8 → 5 allocs。

特に Small の B/op が **56 → 24** に減っているのが `runtime.move2heap()` の効果と考えられます。Go 1.25以前は `cap 1 (8B) + cap 2 (16B) + cap 4 (32B) = 56B` という中間バッキングストアの連なりがヒープ上に作られていましたが、Go 1.26では、ループ中はスタック上の32バイトバッファで処理し、`return` 直前に最終サイズちょうど (`3 要素 × 8B = 24B`) のヒープ領域を1回だけ確保します。中間バッキングストアのガベージが完全に消えているのが読み取れます。

## 4シナリオまとめ

4つのシナリオと、allocs/op がどのバージョンで減るかをまとめます。

| # | シナリオ                         | Small `allocs/op` (1.24 → 1.25 → 1.26) | Large `allocs/op` (1.24 → 1.25 → 1.26) | 鍵になる最適化                        |
| - | -------------------------------- | :------------------------------------: | :------------------------------------: | ------------------------------------- |
| 1 | 定数 `make`                      |              0 → 0 → 0                 |                  —                     | Go 1.24以前で既に最適化済み           |
| 2 | 変数 `make`                      |            1 → **0** → 0               |               1 → 1 → 1                | Go 1.25 の 32B スタックバッファ       |
| 3 | `var` + `append` (非エスケープ) |            3 → **0** → 0               |            8 → **5** → 5               | Go 1.25 の append 箇所スタックバッファ |
| 4 | `var` + `append` + `return`      |            3 → 3 → **1**               |            8 → 8 → **5**               | Go 1.26 の `runtime.move2heap()`      |

このように **Scenario 2 と 3 は同じ Go 1.25 の最適化 (スライスアロケーション箇所への32Bスタックバッファ埋め込み) で allocs が減り、Scenario 4 だけが Go 1.26 の `runtime.move2heap()` で減る**、という分布になります。冒頭の「Go 1.25 と 1.26 で1つずつの最適化」という対応がこの表で裏付けられます。

## もし最適化で問題が起きたら

万が一これらの最適化が原因で、既存のプログラムが想定外の挙動を示してしまった場合の話です。記事中には、以下のフラグで無効化できると記載されています。

```bash
-gcflags=all=-d=variablemakehash=n
```

以前、Go 1.22のfor loop仕様変更に関するブログを書いた際にも、影響範囲を特定するためのフラグ (`-gcflags=all=-d=loopvar=2`) が用意されていましたが、Goチームは **コンパイラの最適化を入れるときに、何かあったときの退避ルートを確実に用意してくれる** 印象があります。

## まとめ

Go 1.25と1.26で段階的に入ったスライスのスタック割り当て最適化について、整理してみました。

個人的に印象的だったのは以下の3点です。

- **開発者がコードを変更することなく、コンパイラが自動的に恩恵をくれる**という点。Goらしい「普通に書けば速い」思想を感じる
- **「Goのスタックフレームは固定サイズ」という制約のもとで、32バイトの小さなバッファを埋め込むというアプローチ**で可変サイズに対応していること。制約の中で工夫している感じがおもしろい
- **`runtime.move2heap()` が手動パターンよりも賢い**こと。手動最適化よりもコンパイラの方がうまくやれるケースがちゃんとある

GoのプロダクトでGCの負荷やアロケーション数を気にされている方にとっては、アップデートの恩恵を自然に受けられる嬉しい変更になりそうです！
